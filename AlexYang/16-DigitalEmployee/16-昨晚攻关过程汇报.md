# 16* 项目群 攻关过程汇报

---

## Prompt 0：终极目标

> **Android APK 无感运行在鸿蒙上。**
>
> - 用户无需知道底层已换成 HarmonyOS
> - 零修改原始 APK，零体验降级
> - 不是"重写一个长得像的鸿蒙 App"，而是"让 .apk 直接跑起来"

这是整个 16* 项目群存在的唯一理由。以下所有子项目，都是为了逼近这个目标而拆解出的攻关路径。

### 诚实评估：16.1 的定位修正

16.1 项目文档自己写得很明确：

> "Android APK **不能直接在鸿蒙上运行**，需要使用 16.1-Adapter 转换代码，重新编译为鸿蒙 .hap 包"

最终上机的 `Index.ets` 是纯 ArkTS 原生代码：
- `@Component struct` 不是 `AppCompatActivity`
- `TextClock()` 是鸿蒙原生组件，不是 Android widget
- `i18n.getTimeZone()` 不是 `java.util.TimeZone`

**16.1 不是 Prompt 0 的实现，而是 Prompt 0 的探路。** 它的价值在于：
1. 验证了 ArkTS 工具链和 DevEco Studio 构建流程
2. 建立了 12 个 Android→HarmonyOS API 映射关系
3. 暴露了"代码翻译"路线的局限 → 推动了 16.4/16.4.1 的启动

**核心产出 — 四层架构**:

| 层 | 名称 | 解决问题 | 可行性 |
|----|------|----------|--------|
| L1 | ART-on-ArkCompiler 兼容层 | JNI、ClassLoader、反射、同步 | ✅ 可行 (6-9月) |
| L2 | 系统服务虚拟化层 | Binder IPC、ContentProvider、Broadcast | ✅ 可行 (3-4月) |
| L3 | 云端服务中继层 | GMS、Firebase、Play Services | ✅ 可行 (3-5月) |
| L4 | 硬件抽象桥接层 | Camera2、USB、NFC、Sensor | ✅ 可行 (4-6月) |

**三大设计原则**:
1. **零体验降级** — 宁可多投入，绝不降级
2. **最小化系统** — 按需实现，禁引入大型 Android 框架
3. **可改 OpenHarmony** — 在不侵权前提下允许修改源码和 ArkCompiler

**8 个代表性 API 深度拆解**:
- R1 `System.loadLibrary()` — JNI 调用链 6 层逐一映射
- R2 `synchronized` 块 — 共享内存线程 → Actor 模型桥接
- R3 `SurfaceView` — 独立渲染面模拟
- R4 `ContentProvider.query()` — 跨进程数据共享虚拟化
- R5 `Android Keystore` — 硬件安全模块桥接
- R6 `Camera2 CaptureRequest` — Pipeline vs Session 适配
- R7 `Parcelable.writeToParcel()` — 序列化协议转换
- R8 `FirebaseMessaging (FCM)` — 推送服务云端中继
