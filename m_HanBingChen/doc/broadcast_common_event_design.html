版本：v1.0 | 日期：2026年3月11日 | 作者：Adapter Team

第一部分：Android 广播机制分析

1.1 核心架构

Android 广播机制基于 AMS（ActivityManagerService） 集中管理，应用通过 Context 发送/接收广播， AMS 负责广播分发、接收者匹配和有序广播的顺序控制。

参与角色

角色类职责
发送方ContextImpl调用 sendBroadcast() / sendOrderedBroadcast()
接收方BroadcastReceiver实现 onReceive(Context, Intent)
注册代理LoadedApk.ReceiverDispatcher包装 BroadcastReceiver 为 IPC Binder 对象
IPC 回调IIntentReceiver.Stub（InnerReceiver）AMS 到应用的回调接口
匹配过滤IntentFilterAction / Category / Data (scheme/host/path/type)
系统服务IActivityManager（AMS）广播注册、匹配分发、有序控制

1.2 IPC 接口方法

以下是 IActivityManager.aidl 中广播相关的全部方法：

方法方向说明
registerReceiverWithFeature(caller, callerPkg, featureId, receiverId, receiver, filter, permission, userId, flags) App → AMS 动态注册广播接收者。receiver 是 IIntentReceiver Binder；filter 是 IntentFilter。返回当前 sticky Intent（如有）
unregisterReceiver(receiver) App → AMS 注销广播接收者。传入注册时的 IIntentReceiver Binder
broadcastIntentWithFeature(caller, featureId, intent, resolvedType, resultTo, resultCode, resultData, map, requiredPermissions, excludePermissions, excludePackages, appOp, options, serialized, sticky, userId) App → AMS 发送广播。serialized=true 为有序广播；sticky=true 为粘性广播；resultTo 为有序广播最终接收者
finishReceiver(who, resultCode, resultData, map, abortBroadcast, flags) App → AMS (oneway) 接收者处理完毕通知 AMS。有序广播中用于传递结果并触发下一个接收者
unbroadcastIntent(caller, intent, userId) App → AMS 撤回粘性广播

回调接口 IIntentReceiver.aidl（oneway）：

oneway interface IIntentReceiver {
    void performReceive(in Intent intent, int resultCode, String data,
            in Bundle extras, boolean ordered, boolean sticky, int sendingUser);
}

1.5 关键特性

第三部分：适配方案设计

3.1 设计目标

  1. Android 应用调用 registerReceiver() 能订阅到 OH 系统事件
  2. Android 应用调用 sendBroadcast() 能发布 OH CommonEvent
  3. OH 系统事件（电池、网络、屏幕等）能以 Android Intent 格式投递给 Android 应用的 BroadcastReceiver
  4. 有序广播、粘性广播语义完整保留
  5. OH 原生应用不受影响

3.4 接口适配详细设计

3.4.1 registerReceiverWithFeature → CES.SubscribeCommonEvent

当 Android 应用调用 registerReceiver(receiver, filter) 时：

  1. IntentFilter → MatchingSkills 转换 
    // IntentFilter 中的每个 Action
for (action : filter.actions) {
    String ohEvent = ActionMapper.androidToOH(action);
    matchingSkills.AddEvent(ohEvent);
}
// Category → Entity
for (category : filter.categories) {
    matchingSkills.AddEntity(category);
}
// Data Scheme
for (scheme : filter.dataSchemes) {
    matchingSkills.AddScheme(scheme);
}
  2. 创建订阅信息 
    CommonEventSubscribeInfo subscribeInfo(matchingSkills);
subscribeInfo.SetPriority(filter.getPriority());
if (requiredPermission != null) {
    subscribeInfo.SetPermission(requiredPermission);
}
  3. JNI 调用订阅：传递序列化的 MatchingSkills + 订阅 ID 到 C++ 层
  4. C++ 层创建 CommonEventSubscriber：保存订阅 ID 和 IIntentReceiver 的 JNI 引用
  5. 注册到 CES：CommonEventManager::SubscribeCommonEvent(subscriber)
  6. 返回 sticky Intent：如果有匹配的 sticky 事件，查询 GetStickyCommonEvent() 并转换返回

3.4.2 broadcastIntentWithFeature → CES.PublishCommonEvent

当 Android 应用调用 sendBroadcast(intent) 时：

  1. Intent → CommonEventData 转换 
    // Action 映射
String ohAction = ActionMapper.androidToOH(intent.getAction());
Want want;
want.SetAction(ohAction);

// Extras → Want Params
if (intent.getExtras() != null) {
    String json = bundleToJson(intent.getExtras());
    want.SetParam("android_extras_json", json);
}

// Data URI → Want URI
if (intent.getData() != null) {
    want.SetUri(intent.getData().toString());
}

CommonEventData eventData;
eventData.SetWant(want);
eventData.SetCode(resultCode);
eventData.SetData(resultData);
  2. 发布参数转换 
    CommonEventPublishInfo publishInfo;
publishInfo.SetOrdered(serialized);   // ordered broadcast
publishInfo.SetSticky(sticky);        // sticky broadcast
if (requiredPermissions != null) {
    publishInfo.SetSubscriberPermissions(requiredPermissions);
}
  3. JNI 调用发布
  4. C++ 层调用：CommonEventManager::PublishCommonEvent(eventData, publishInfo)

3.4.3 OH CommonEvent → IIntentReceiver.performReceive (反向回调)

当 CES 投递事件给 Adapter 的 CommonEventSubscriber 时：

  1. C++ OnReceiveEvent 回调触发：Adapter 的 CommonEventSubscriber 子类收到事件
  2. CommonEventData → Intent 转换 
    // Action 映射
String androidAction = ActionMapper.ohToAndroid(want.GetAction());
Intent intent = new Intent(androidAction);

// Want params → Bundle extras
String json = want.GetStringParam("android_extras_json");
if (json != null) {
    intent.putExtras(jsonToBundle(json));
}

// Want URI → Intent data
if (!want.GetUri().empty()) {
    intent.setData(Uri.parse(want.GetUri()));
}
  3. JNI 回调到 Java：调用 CommonEventReceiverBridge.onCommonEventReceived(subscriptionId, intent, code, data, ordered, sticky)
  4. Java 层分发到 IIntentReceiver 
    IIntentReceiver receiver = mReceiverMap.get(subscriptionId);
receiver.performReceive(intent, resultCode, resultData,
    extras, ordered, sticky, sendingUser);

3.4.4 finishReceiver → CES.FinishReceiver

有序广播中，当 BroadcastReceiver 处理完毕调用 PendingResult.finish() 时：

  1. AMA 收到 finishReceiver(who, resultCode, resultData, map, abortBroadcast)
  2. 通过 JNI 调用 C++ nativeFinishCommonEvent(subscriptionId, resultCode, resultData, abortEvent)
  3. C++ 层调用 AsyncCommonEventResult::FinishCommonEvent() 或设置结果并通知 CES

3.7 有序广播适配

有序广播是适配的难点，因为 Android 和 OH 的有序机制存在差异：

差异点AndroidOH
完成通知finishReceiver() IPC 到 AMSAsyncCommonEventResult 或自动完成
中止机制abortBroadcast()AbortCommonEvent()
结果传递resultCode + resultData + resultExtrascode + data（无 extras 链式传递）
最终接收者sender 提供 resultReceiver不直接支持

3.7.1 有序广播处理方案

核心思路：Adapter 层在 C++ 端使用 OH 有序事件机制（GoAsyncCommonEvent），在完成通知路径上桥接 Android 的 finishReceiver 语义。

订阅有序事件：

  • C++ AdapterEventSubscriber 在 OnReceiveEvent() 中：
    • 检测 ordered == true，调用 GoAsyncCommonEvent() 获取 AsyncCommonEventResult
    • 保存 AsyncCommonEventResult 到 mOrderedBroadcastState[subscriptionId]
    • 回调 Java 层时传递 ordered=true

finishReceiver 桥接：

  • Java 层 finishReceiver(who, resultCode, resultData, map, abortBroadcast) 被调用
  • 查找 who (IBinder) → subscriptionId
  • JNI 调用 nativeFinishCommonEvent(subscriptionId, resultCode, resultData, abortEvent)
  • C++ 层从 mOrderedBroadcastState 取出 AsyncCommonEventResult
  • 设置 code/data，若 abort 则调用 AbortCommonEvent()
  • 调用 AsyncCommonEventResult::FinishCommonEvent()，CES 继续投递下一个订阅者

最终接收者（resultReceiver）：

  • Android 发送有序广播时的 resultTo（IIntentReceiver）在 Adapter 层保存
  • Adapter 发布有序事件时附带一个自身的 CommonEventListener 作为"最终回调"
  • CES 有序分发全部完成后，回调此 listener，Adapter 再调用 resultTo.performReceive()

3.8 粘性广播适配

场景处理方式
sendStickyBroadcast(intent) 调用 PublishCommonEvent() 时设置 publishInfo.SetSticky(true)
registerReceiver() 返回 sticky Intent 注册完成后调用 GetStickyCommonEvent(event) 获取缓存事件，转换为 Intent 返回
removeStickyBroadcast(intent) 调用 RemoveStickyCommonEvent(action)

3.9 权限适配

Android 权限场景OH 适配
发送广播时要求接收方权限：sendBroadcast(intent, permission) publishInfo.SetSubscriberPermissions({permission})
注册时要求发送方权限：registerReceiver(rcvr, filter, permission, handler) subscribeInfo.SetPermission(permission)
权限名映射：Android 权限名（如 android.permission.RECEIVE_SMS）需要映射到 OH 权限名 （如 ohos.permission.RECEIVE_SMS）。无法直接映射的权限，由 Adapter 层在回调时自行做权限过滤。

3.10 自定义广播（App-to-App）适配

Android 应用之间的自定义广播（非系统广播）也需要支持：

  • 发送：自定义 Action（如 com.example.MY_ACTION）映射为 OH 事件名 adapter.custom.com.example.MY_ACTION
  • 订阅：IntentFilter 中的自定义 Action 同样加 adapter.custom. 前缀后订阅到 CES
  • 限制：仅适配层内的 Android 应用之间可互通自定义广播。OH 原生应用不会收到这些事件（除非显式订阅带 adapter.custom. 前缀的事件）

3.11 Intent extras 与 Want params 转换

Android Intent extras（Bundle）与 OH Want params 的数据格式不同，通过 JSON 序列化桥接：

// Android → OH（发送方）
Bundle extras = intent.getExtras();
String json = BundleJsonConverter.toJson(extras);
want.SetParam("android_extras_json", json);

// OH → Android（接收方）
String json = want.GetStringParam("android_extras_json");
Bundle extras = BundleJsonConverter.fromJson(json);
intent.putExtras(extras);

对于已知系统广播的特定 extras key，进行原生 Want param 映射：

// 例如 BATTERY_CHANGED
// Android extras: "level"=80, "scale"=100, "status"=2
// OH Want params: want.SetParam("level", 80), want.SetParam("scale", 100), ...
// 通过 EventDataMapper 类根据 action 做专门映射

第八部分：风险与缓解

风险影响缓解措施
IntentFilter 的 Data Host/Path/Type 匹配在 OH MatchingSkills 中不支持 部分精细过滤失效，接收者收到不应收到的事件 Adapter 层在 Java 回调路径上做二次过滤：收到事件后检查完整 IntentFilter 匹配，不匹配则丢弃
系统事件 extras 格式差异 Android 应用读取 extras 时 key/type 不匹配 对高频系统事件（BATTERY_CHANGED、CONNECTIVITY_CHANGE 等）编写专用 EventDataMapper，精确转换每个 extra 字段
Android 权限名与 OH 权限名不同 权限过滤可能不生效 维护权限映射表；无法映射时在 Adapter 层做本地权限检查
有序广播 resultExtras (Bundle) 链式传递 OH 有序事件仅支持 code + data 传递，不支持 extras 链式修改 Adapter 层本地维护 extras 传递状态。每个接收者的 setResultExtras 保存在 Adapter 的 OrderedBroadcastState 中，下一个接收者的 performReceive 使用上一个的 extras
Manifest 静态注册的 BroadcastReceiver Android 应用的 AndroidManifest.xml 中声明的静态 receiver 不经过 registerReceiver() Adapter 启动时解析 AndroidManifest，提取静态 receiver 声明，自动注册到 CES。或利用 OH StaticSubscriberExtension 机制
大量 Android 应用注册相同 Action 的性能 每个 registerReceiver 都产生一个 CES 订阅 Adapter 层合并相同 Action 的订阅：多个 IIntentReceiver 共享一个 CES 订阅，由 Adapter 做本地分发
