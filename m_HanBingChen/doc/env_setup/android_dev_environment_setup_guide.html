1. 需求分析与组件规划

1.1 项目运行时全景图

本项目的目标是在 OpenHarmony 系统上运行 Android App。运行时的完整栈如下：

3.1 部分同步策略

AOSP 完整源码包含 600+ 个 Git 仓库，我们只需要其中约 30-40 个。策略：

  1. 使用 repo init 获取完整 manifest（仅 manifest，不下载代码）
  2. 使用 repo sync 指定项目名，只下载需要的仓库
  3. 对于缺少的依赖，按需追加同步

4. 组件 A：AIDL Stub 一次性生成

4.1 目标与原理

适配层的核心模式是 class inheritance：每个 Adapter 类继承 AIDL 自动生成的 IXxx.Stub 类。例如：

// ActivityManagerAdapter.java
public class ActivityManagerAdapter extends IActivityManager.Stub {
    // 必须覆写 IActivityManager.aidl 中定义的所有 249 个方法
    @Override
    public int startActivity(...) { ... }
    ...
}

IActivityManager.Stub 是 AIDL 编译器从 IActivityManager.aidl 自动生成的抽象类。我们需要：

  1. 用 AOSP 16 的 AIDL 编译器处理所有 .aidl 文件
  2. 生成 Java 源文件（IActivityManager.java 等）
  3. 将生成结果保存到适配层项目中
  4. 后续手动维护，不再依赖 AIDL 编译器自动生成

4.2 需要生成的 AIDL 接口清单

主要接口（适配层直接继承）

AIDL 的级联依赖：一个 AIDL 文件 import 其他 AIDL 和 Parcelable，因此不能只编译单个文件。需要编译整个 frameworks/base 的 AIDL 集合，AOSP 构建系统会自动处理依赖。

4.3 方法一：使用 AOSP 构建系统生成（推荐）

cd ~/aosp16

# 初始化编译环境
source build/envsetup.sh
lunch aosp_arm64-userdebug

# 只编译 framework 模块的 AIDL → Java（不编译完整 framework）
# 触发 AIDL 生成的最小编译目标：
m framework-minus-apex-headers -j$(nproc) 2>&1 | tee build_aidl.log

# 或直接编译 framework 模块（会生成所有 AIDL Stub）
m framework -j$(nproc) 2>&1 | tee build_framework.log

查找生成的 AIDL Java 文件

# AIDL 生成的 Java 文件位于 Soong 中间产物目录
AIDL_OUT="out/soong/.intermediates/frameworks/base"

# 查找 IActivityManager.java
find ${AIDL_OUT} -name "IActivityManager.java" 2>/dev/null

# 查找所有生成的 AIDL Java 文件
find ${AIDL_OUT} -name "*.java" -path "*/aidl/*" 2>/dev/null | head -30

# 典型路径示例：
# out/soong/.intermediates/frameworks/base/framework/android_common/
#   gen/aidl/frameworks/base/core/java/android/app/IActivityManager.java

4.4 方法二：独立运行 AIDL 编译器

如果不想触发完整的 Soong 编译流程，可以直接使用 AIDL 编译器二进制：

# AIDL 编译器位于 prebuilts 中
AIDL_BIN=$(find ~/aosp16/prebuilts -name "aidl" -type f -executable 2>/dev/null | head -1)
echo "AIDL compiler: ${AIDL_BIN}"

# AIDL 源文件根目录
FRAMEWORK_AIDL="~/aosp16/frameworks/base/core/java"

# 输出目录
AIDL_GEN_OUT="/tmp/aidl_gen"
mkdir -p ${AIDL_GEN_OUT}

# 生成 IActivityManager.java
${AIDL_BIN} \
    --lang=java \
    -I ${FRAMEWORK_AIDL} \
    -I ~/aosp16/frameworks/base/media/java \
    -I ~/aosp16/frameworks/base/location/java \
    -I ~/aosp16/frameworks/base/telephony/java \
    -I ~/aosp16/frameworks/native/aidl/binder \
    -I ~/aosp16/frameworks/native/aidl/gui \
    -o ${AIDL_GEN_OUT} \
    ${FRAMEWORK_AIDL}/android/app/IActivityManager.aidl

# 对其他 AIDL 文件重复执行
# 注意：-I 参数需要包含所有被 import 的 AIDL 路径
方法二的缺点：需要手动解决 import 路径（-I 参数），比较繁琐。方法一（AOSP 构建系统）自动处理所有依赖关系，推荐优先使用。

4.6 后续维护策略

AIDL 生成文件的维护规则

  1. 一次性生成：从 AOSP 16 的 AIDL 编译器生成完整的 Stub Java 文件
  2. 纳入版本管理：将生成的文件提交到 adapter Git 仓库
  3. 手动维护：后续如需修改方法签名、增减方法，直接编辑 Java 文件
  4. 不再自动生成：Android.bp 中 adapter 的 Java 编译直接引用这些文件，而非 .aidl
  5. 标注来源版本：在文件头部注释中标注 Generated from AOSP 16 (android-16.0.0_r1)
何时需要重新生成：仅当升级 AOSP 基准版本时（如从 AOSP 16 → 17），需要重新生成 AIDL Stub 并 diff 对比变化，合并到手动修改版本中。

5. 组件 B：ART 虚拟机编译

5.1 ART 在本项目中的角色

ART（Android Runtime）是运行 Android App 的核心引擎。在 DAYU600 (OH) 上运行 Android App 必须部署 ART：

  • DEX 解释执行：ART 解释器执行 APK 中的 DEX 字节码
  • JIT 编译：运行时将热点代码编译为 ARM64 本地代码
  • GC 管理：管理 Android App 的 Java 堆内存
  • JNI 桥接：支持 Java ↔ Native 方法调用
  • appspawn-x 依赖：appspawn-x 通过 JniInvocation 加载 libart.so 创建 VM 实例

5.2 ART 构建目标与产物

构建目标产物说明
libartlibart.soART 运行时核心（解释器、GC、JNI）
libart-compilerlibart-compiler.soJIT/AOT 编译器后端
libdexfilelibdexfile.soDEX 文件解析
libartbaselibartbase.soART 基础工具库
libprofilelibprofile.soProfile 引导优化
dalvikvmdalvikvm (binary)VM 启动入口（可选，appspawn-x 替代）

5.4 ART 运行时依赖链

libart.so 运行时依赖以下共享库，需要一并编译和部署：

libart.so 依赖树（主要依赖） libart.so ├── libdexfile.so (DEX 文件操作) ├── libartbase.so (ART 基础) ├── libprofile.so (Profile) ├── libbase.so (Android base) ├── liblog.so (日志) ├── libz.so (zlib 压缩) ├── liblz4.so (lz4 压缩) ├── libunwindstack.so (栈回溯) ├── libc++.so (C++ 标准库) ├── libdl.so (动态链接) ├── libc.so (Bionic C 库) └── libm.so (数学库) libart-compiler.so ├── libart.so ├── libvixl.so (ARM 汇编器) └── ... dex2oat ├── libart-compiler.so ├── libart.so └── ...
Bionic 兼容性问题：ART 依赖 Android Bionic (libc.so)，而 OH 系统使用 musl libc。两者 ABI 不兼容。解决方案：
  1. 方案 A：将 Bionic libc/libdl/libm 一同部署，放在独立路径（如 /system/android/lib64/），通过 LD_LIBRARY_PATH 或 linker namespace 隔离
  2. 方案 B：将 ART 静态链接 Bionic（工程量大，不推荐）
  3. 方案 C：使用 linker namespace（Android 的 linker 本身就支持命名空间隔离）
推荐方案 A，最简单可行。

6. 组件 C：dex2oat 编译

6.1 dex2oat 在本项目中的角色

dex2oat 是 ART 的 AOT（Ahead-Of-Time）编译器，在 APK 安装时将 DEX 字节码编译为本地机器码：

APK 安装流程中 dex2oat 的位置 APK 文件 (classes.dex) ↓ apk_installer（OH 编译产物） ↓ 调用 dex2oat --dex-file=classes.dex \ --oat-file=base.odex \ --instruction-set=arm64 \ --boot-image=/system/android/framework/boot.art ↓ base.odex + base.vdex（优化后的本地代码） ↓ App 启动时 ART 直接加载 .odex（无需再解释执行 DEX）

参见 framework/package-manager/jni/apk_installer.h 第 63 行：dex2oat 路径为 /system/android/bin/dex2oat

6.2 编译 dex2oat

dex2oat 是 ART 的一部分，在组件 B 的编译中已包含：

# 已在组件 B 步骤中编译，确认产物存在：

7. 组件 D：Framework Boot Classpath

7.1 Boot Classpath 的作用

Android App 运行前，ART VM 需要加载 Boot Classpath——一组核心 JAR 文件，提供 java.lang.*、android.app.*、android.view.* 等基础类。这些 JAR 对应 appspawn-x 中的 BOOTCLASSPATH 环境变量。

Boot Classpath 中的类被预编译为 Boot Image（boot.art + boot.oat），ART 启动时直接映射到内存，大幅加速启动速度。

7.2 Boot Classpath JAR 清单

JAR 文件AOSP 来源内容
core-oj.jarlibcore/ojluni/OpenJDK 核心类（java.lang, java.util, java.io 等）
core-libart.jarlibcore/libart/ART 专有核心类（dalvik.system 等）
conscrypt.jarexternal/conscrypt/TLS/SSL 加密 provider
okhttp.jarexternal/okhttp/HTTP 客户端
bouncycastle.jarexternal/bouncycastle/加密算法实现
apache-xml.jarexternal/apache-xml/XML 解析
framework.jarframeworks/base/Android Framework 核心（含 AOSP patch + AIDL Stub）
ext.jarframeworks/base/Framework 扩展
icu4j.jarexternal/icu/国际化支持
framework.jar 是关键：它包含了 IActivityManager.Stub 等 AIDL 生成类，以及 ActivityManager、WindowManagerGlobal 等被 AOSP patch 修改的类。修改 AOSP patch 后需要重新编译 framework.jar。

9. appspawn-x 与 Zygote

9.1 appspawn-x 替代 Zygote

标准 Android 使用 Zygote 进程孵化 App 进程（frameworks/base/cmds/app_process/）。本项目使用 appspawn-x——一个混合 OH appspawn + Android Zygote 的进程孵化器，替代标准 Zygote：

功能标准 Android Zygoteappspawn-x
进程管理Zygote forkOH appspawn fork
安全策略Android SELinuxOH 沙箱 + Android 兼容
VM 创建AndroidRuntime::start()AppSpawnXRuntime (libart.so)
JNI 注册register_jni_procs()相同（通过 libandroid_runtime）
类预加载ZygoteInit.preload()AppSpawnXInit.preload()

9.2 appspawn-x 对 ART 的依赖

appspawn-x 的 BUILD.gn 引用以下 ART/Android 预编译库：

# 来自 framework/appspawn-x/BUILD.gn
deps = [
    "//third_party/android/runtime:libandroid_runtime",  # AndroidRuntime, JNI 注册
    "//third_party/android/runtime:libnativehelper",      # JniInvocation
    "//third_party/android/art:libart",                   # ART VM
    "//third_party/android/art:libdexfile",               # DEX 文件处理
]

这意味着：AOSP 编译 ART 后，需要将产物复制到 OH 源码树的 third_party/android/ 目录下，作为预编译库供 appspawn-x 链接。

9.3 appspawn-x 编译（OH GN 构建系统）

# 在 OH 编译服务器上（不是 AOSP 编译环境）
cd ~/openharmony

# 步骤 1：准备 ART 预编译库
# 将 AOSP 编译的 ART 产物复制到 OH 源码树
mkdir -p third_party/android/art/prebuilt/arm64
mkdir -p third_party/android/runtime/prebuilt/arm64

# 从 AOSP 产物目录复制（或通过 rsync 从远程下载）
cp libart.so libart-compiler.so libdexfile.so libartbase.so \
    third_party/android/art/prebuilt/arm64/
cp libnativehelper.so libandroid_runtime.so \
    third_party/android/runtime/prebuilt/arm64/

# 步骤 2：编译 appspawn-x
./build.sh --product-name dayu210 --ccache \
    --build-target appspawn_x 2>&1 | tee build_appspawnx.log

10.2 一键部署脚本

#!/bin/bash
# deploy_all.sh — 部署所有 AOSP 编译产物到 DAYU600
AOSP_ARTIFACTS="/mnt/d/code/adapter/build/aosp_artifacts"

# 检查设备
DEVICE=$(hdc list targets 2>/dev/null | head -1)
[ -z "$DEVICE" ] || [ "$DEVICE" = "[Empty]" ] && echo "未检测到设备" && exit 1
echo "设备: $DEVICE"

# 挂载可写
hdc shell mount -o remount,rw /

# 创建目录结构
hdc shell mkdir -p /system/android/lib64
hdc shell mkdir -p /system/android/bin
hdc shell mkdir -p /system/android/framework
hdc shell mkdir -p /system/app/android/OHAdapterHelloWorld

echo "===== 推送 ART 库 ====="
for so in libart.so libart-compiler.so libdexfile.so libartbase.so \
          libnativehelper.so libbase.so liblog.so libutils.so libcutils.so \
          libc++.so libz.so liblz4.so libunwindstack.so libvixl.so \
          libc.so libm.so libdl.so; do
    [ -f "${AOSP_ARTIFACTS}/${so}" ] && \
        hdc file send "${AOSP_ARTIFACTS}/${so}" "/system/android/lib64/${so}" && \
        echo "  ${so}"
done

echo "===== 推送 dex2oat ====="
hdc file send "${AOSP_ARTIFACTS}/dex2oat" /system/android/bin/dex2oat
hdc shell chmod 755 /system/android/bin/dex2oat

echo "===== 推送 Boot Classpath ====="
for jar in core-oj.jar core-libart.jar conscrypt.jar framework.jar ext.jar \
           oh-adapter-framework.jar; do
    [ -f "${AOSP_ARTIFACTS}/${jar}" ] && \
        hdc file send "${AOSP_ARTIFACTS}/${jar}" "/system/android/framework/${jar}" && \
        echo "  ${jar}"
done

echo "===== 推送 Boot Image ====="
[ -f "${AOSP_ARTIFACTS}/boot.art" ] && \
    hdc file send "${AOSP_ARTIFACTS}/boot.art" /system/android/framework/boot.art
[ -f "${AOSP_ARTIFACTS}/boot.oat" ] && \
    hdc file send "${AOSP_ARTIFACTS}/boot.oat" /system/android/framework/boot.oat

echo "===== 推送适配层 JNI ====="
[ -f "${AOSP_ARTIFACTS}/liboh_adapter_bridge.so" ] && \
    hdc file send "${AOSP_ARTIFACTS}/liboh_adapter_bridge.so" /system/lib64/liboh_adapter_bridge.so

echo "===== 推送测试 APK ====="
[ -f "${AOSP_ARTIFACTS}/OHAdapterHelloWorld.apk" ] && \
    hdc file send "${AOSP_ARTIFACTS}/OHAdapterHelloWorld.apk" \
        /system/app/android/OHAdapterHelloWorld/OHAdapterHelloWorld.apk

echo ""
echo "===== 部署完成，重启设备 ====="
hdc shell reboot

12. 注意事项与常见问题

问题原因解决方案
部分 sync 后编译报缺少模块 有依赖的 AOSP 项目未同步 根据报错信息追加同步：repo-google sync -c --depth=1 <project>
lunch 找不到 target device/ 目录未同步 追加同步 device/generic/common 和 device/generic/arm64

OH repo 和 AOSP repo 冲突 使用了错误的 repo 工具 OH: /usr/local/bin/repo
AOSP: /usr/local/bin/repo-google
ccache 缓存冲突 OH 和 AOSP 共用 ccache 分开：CCACHE_DIR=/root/.ccache-aosp
AIDL 生成的方法数量与文档不同 AOSP 16 vs AOSP 14 接口有变更 以 AOSP 16 生成结果为准，更新适配层 Adapter 类和 CLAUDE.md 中的统计
framework.jar 编译拉取大量依赖 framework 模块依赖链较深 按报错追加同步缺失项目；首次可能需要多轮
