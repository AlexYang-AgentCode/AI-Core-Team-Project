1. 项目背景与目标

本报告旨在评估将 Android Framework 与 OpenHarmony 系统服务进行对接的技术可行性。核心构想是在 Android Framework 中增加一个适配层（Adaptation Layer），实现以下两个目标：

  1. App → 系统服务方向：将 Android App 作为 Client 调用系统服务的接口，转换为调用 OpenHarmony 系统服务的接口；
  2. 系统服务 → App 方向：将 OpenHarmony 系统服务调用 App 的接口，转换为 Android App 能够理解的接口。

两个系统均采用 Binder 机制进行跨进程 IPC 通信，但其接口定义、数据结构和底层实现存在显著差异。

分析范围

本报告对 Android 应用调用系统服务的 4 个正向接口（共 530 个方法）和 OH 系统回调 Android 应用的 6 个反向接口（共 181+ 个方法）进行了逐方法级别的可行性分析，同时从 Android 回调接口视角分析了 47 个方法的覆盖缺口。采用三级分类：

  • 可映射 — 存在 OH 对应接口，可通过直接映射或逻辑转换完成适配
  • 需OH增强 — 无 OH 对应接口，需 OH 系统服务新增能力
  • 可忽略 — 可安全忽略（返回默认值/空操作），不影响 App 正常运行

2.3 生命周期状态机差异

Android Activity:
  onCreate → onStart → onResume → onPause → onStop → onDestroy

OpenHarmony Ability (Stage模型):
  onCreate → onWindowStageCreate → onForeground → onBackground → onWindowStageDestroy → onDestroy

关键差异：OpenHarmony 的 Ability 生命周期与 WindowStage 紧耦合，而 Android 的 Activity 生命周期与 Window 相对松耦合。适配层需要在两种状态机之间建立正确的映射关系。

P2 可选4项8个方法6.5人周 备份代理（3人周）、壁纸回调（2人周）、兼容性（0.5人周）、动画回调（1人周）
P3 延后5项24个方法11.5人周 语音交互（5人周）、证书（0.5人周）、滚动截图（2人周）、调试诊断（3人周）、其他IWindow（1人周）
合计16项47个方法32人周 P0+P1（核心）：14人周；P2+P3（完整）：18人周

核心结论

47个Android回调方法（60%）缺少OH回调接口对应，但其中：

  • 11个高影响方法（P0）必须由OH系统服务增强能力提供，否则广播接收、Service运行、ContentProvider等核心功能无法工作
  • 8个中影响方法（P1）建议OH尽快支持，影响系统设置同步和网络安全
  • 8个可选方法（P2）可根据项目进度安排
  • 20个低影响方法（P3）可安全忽略或延后

OH侧核心改造集中在IAppScheduler（新增约6个方法）和IAbilityScheduler（新增约2个方法）两个接口，预计P0+P1工作量约14人周。

8. 核心技术挑战

8.1 Binder 跨系统互通

  • Android Binder 和 OH IPC 框架是两套独立的内核驱动和用户态框架
  • Android 使用 AIDL 生成 Java Proxy/Stub；OH 使用 C++ IRemoteProxy/IRemoteStub
  • 必须解决：两个 Binder 域之间的消息转发机制
  • 方案：在 App 进程内通过 JNI 桥接，Java 层拦截 Android Binder 调用，转入 C++ 层使用 OH IPC 框架发起跨进程调用

8.2 Intent ↔ Want 转换

  • Android Intent (Java)：action / category / data(URI) / extras(Bundle) / component
  • OH Want (C++)：bundleName / abilityName / action / entities / parameters
  • 需要建立完整的映射规则和转换层，尤其是隐式 Intent 的解析

8.3 图形渲染管道

  • Android：SurfaceFlinger + BufferQueue + HWC
  • OH：RenderService + Surface + HWC
  • 这是最大的难点，窗口合成和显示是底层紧耦合的
  • 需要共享 GraphicBuffer 或建立 Surface 代理层

8.4 输入系统

  • Android：InputManagerService → InputChannel (socketpair) → App
  • OH：多模输入子系统 → Session → App
  • 输入事件格式（MotionEvent vs. PointerEvent）和分发路径完全不同

8.5 运行环境共存

  • Android App 进程运行在 ART (Android Runtime) 上
  • OH IPC 客户端库是 C++ 实现
  • 需要确保同一进程中可同时加载 ART 和 OH IPC 动态库

8.6 技术挑战优先级排序

优先级挑战影响范围难度建议方案
P0Intent ↔ Want转换所有Activity/Service启动 高 完善IntentWantConverter，建立Action映射表，处理extras序列化
P0窗口Surface共享所有UI渲染 很高 研究OH RenderService共享内存机制，建立Surface桥接
P0OH细粒度→Android批量回调聚合IWindow-OH/ISessionStage回调 高 引入缓冲+定时/条件触发的回调聚合机制
P1生命周期状态映射Activity生命周期 中 Android 6态→OH 4态映射已设计，需处理反向映射歧义
P1ClientTransaction构造AbilityScheduler→IApplicationThread 高 根据OH Ability状态变化构造对应的ClientTransaction
P2广播系统重新架构11+个方法 很高 OH CommonEventManager→Android BroadcastReceiver转换层
P2ContentProvider数据格式10+个方法 高 NativeRdb.ValuesBucket↔ContentValues、DataAbilityPredicates↔SQL转换器
P3权限模型适配18+个方法 中 OH与Android权限字符串映射，运行时权限流程桥接

9.2 适配层核心模块设计

模块1：LifecycleAdapter（生命周期适配）

App → 系统服务方向

ActivityManager.startActivity(Intent)
  → LifecycleAdapter.convertIntent(Intent) → Want
  → [JNI Bridge]
  → AbilityManagerProxy.StartAbility(Want)

系统服务 → App 方向

IAppScheduler.ScheduleLaunchAbility()
  → [JNI Bridge]
  → LifecycleAdapter.convertToClientTransaction()
  → IApplicationThread.scheduleTransaction(ClientTransaction)

模块2：WindowAdapter（窗口适配）

App → 系统服务方向

ViewRootImpl → IWindowSession.addToDisplay()
  → WindowSessionAdapter.addToDisplay()
  → [JNI Bridge]
  → ISceneSessionManager.CreateAndConnectSpecificSession() (SA 4607)
  → 返回 ISession 每窗口代理（存储供后续使用）

ViewRootImpl → IWindowSession.relayout()
  → WindowSessionAdapter.relayout()
  → [JNI Bridge]
  → ISession.UpdateSessionRect()  (每窗口代理)

系统服务 → App 方向

ISessionStage.UpdateRect()
  → [JNI Bridge] → SessionStageBridge
  → IWindow.resized()

模块3：IPC Bridge（跨 Binder 域桥接）

核心设计思路：

  1. 拦截点：在 Android Framework 的 ServiceManager.getService() 处拦截，返回适配层的 Proxy 替代原 SystemServer 的 Binder
  2. JNI 层：Java Proxy 通过 JNI 调用 C++ 适配代码
  3. OH IPC 调用：C++ 层使用 OH 的 IRemoteProxy 发起真正的跨进程调用
  4. 反向回调：C++ 层实现 OH 的 IRemoteStub 接收系统服务回调，通过 JNI 调回 Java 层

10.3 风险评估

风险提示

  • Surface共享机制尚未验证可行性，可能需要OH RenderService侧的协助，是最大技术风险
  • 广播系统重新架构涉及OH CommonEventManager和Android BroadcastReceiver两个不同模型的桥接，复杂度高
  • AOSP版本跟随：AOSP源码补丁需要随Android版本更新维护，长期成本需考虑
  • 性能开销：JNI桥接和数据格式转换（特别是Intent↔Want、ContentValues↔ValuesBucket）的性能影响需要评估
  • OH系统服务改造依赖（第7章）：广播投递、Service创建/销毁、ContentProvider生命周期3项P0增强需要OH系统服务（IAppScheduler/IAbilityScheduler）新增接口方法，涉及OH侧AbilityManagerService和AppMgrService代码修改，需与OH系统团队协调

12. 结论

技术上有条件可行，工程量可控。核心判断如下：

  1. 接口映射可行性高：711个方法中，43%可直接映射，43%可安全忽略，仅14%需OH增强。实际需适配的408个方法中75%可映射。Class继承模式（extends IXxx.Stub）的桥接架构已验证可行。
  2. 生命周期和服务管理层的适配是可行的。两套系统的概念模型足够相似（Activity ↔ Ability, Service ↔ ServiceExtension, Intent ↔ Want），可以建立合理的映射关系。
  3. 图形/窗口系统是最大瓶颈。Surface 和输入通道是底层紧耦合的，需要 kernel 层面的协调或图形缓冲区共享机制。
  4. OH系统服务增强是可控的。47个缺口方法中，P0+P1核心增强仅需14人周OH侧改造，集中在IAppScheduler和IAbilityScheduler两个接口。
  5. 前提条件：Android App 进程需要能同时加载 Android Runtime (ART) 和 OH IPC 动态库，即运行环境需要支持两套框架共存。这要求底层 Linux 内核同时加载两个 Binder 驱动（或 OH 使用用户态 IPC 方案）。
  6. 工作量估算：核心适配约63.6人周（~15.9人月），完整适配约116.6人周（~29.2人月），分4个阶段实施。
  7. 建议：先用阶段一做最小 POC 验证，打通一条完整的调用链路（startActivity → Ability 启动 → 前台回调），再决定是否全面推进。

该方案的本质类似于 Android 兼容层（如 Windows Subsystem for Android 的逆向思路），在工业界有类似先例，但需要大量底层系统级开发工作。
