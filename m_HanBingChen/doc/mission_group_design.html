Android Task → OH Mission 多 Ability 栈适配设计方案

版本：v2.0 | 日期：2026年3月11日 | 作者：Adapter Team

v2.0 变更：废弃 MissionGroup 方案，改为 Mission 内 Ability 栈方案。修改集中在 Mission 类内部， 无需修改 MissionInfo/InnerMissionInfo/MissionInfoMgr/IMissionManager 等外层接口。

1. 问题分析

1.1 核心差异

概念AndroidOpenHarmony
任务容器 Task：包含多个 ActivityRecord 的栈 Mission：仅包含一个 AbilityRecord
打开新页面 新 Activity 入栈到当前 Task 创建新 Mission + 新 Ability
最近任务列表 每个 Task 一条，显示栈顶 Activity 每个 Mission 一条，显示其唯一 Ability
后退/返回 弹出栈顶 Activity，回到栈内前一个 销毁当前 Mission，回到前一个 Mission

1.2 设计目标

  1. OH 原生应用：行为完全不变，Mission 内始终只有 1 个 Ability
  2. Android 适配应用：同一应用的多个 Activity 共用同一个 Mission，形成 Ability 栈
  3. 最近任务列表：每个 Mission 显示栈顶 Ability，Android 应用自然只显示一条
  4. 后退行为：弹出栈顶 Ability 后露出前一个 Ability，栈空时销毁 Mission
  5. OH Launcher 无需修改：GetMissionInfos 接口语义不变
  6. IMissionManager 接口无需扩展：所有变更封闭在 Mission 类内部

2. 方案对比

v1.0 MissionGroup 方案v2.0 Mission 内 Ability 栈方案
核心思路 在 Mission 之上引入 MissionGroup 分组层 将 Mission 的 abilityRecord_ 扩展为 Ability 栈
MissionInfo 修改 需增加 groupId, groupSize, isGroupTop 字段 无需修改
InnerMissionInfo 修改 需增加 groupId, groupOrder, isAndroidApp 字段 无需修改
MissionInfoMgr 修改 需增加 GetGroupedMissionInfos 等 6 个方法 无需修改
IMissionManager 接口 需增加 6 个 IPC 方法 + IPC code 无需修改
OH Launcher 需修改以使用新接口或过滤 isGroupTop 无需修改
修改范围 8 个 OH 系统文件 + 6 个 Adapter 文件 3 个 OH 系统文件 + 3 个 Adapter 文件
对 OH 系统的侵入性 高（接口扩展、数据结构修改、持久化修改） 低（仅 Mission 类内部实现变更）

3. 总体架构

3.1 核心概念：Mission 内 Ability 栈

【OH 原生应用 — 行为不变】 【Android 适配应用 — Ability 栈】 ┌─────────────────────────┐ ┌─────────────────────────┐ │ Mission (id=1) │ │ Mission (id=2) │ │ │ │ │ │ abilityStack_: │ │ abilityStack_: │ │ ┌─────────────────────┐ │ │ ┌─────────────────────┐ │ │ │ WeatherAbility │ │ │ │ ThirdActivity ←─┼─── 栈顶 (top) │ │ (唯一, 即栈顶也是底) │ │ │ ├─────────────────────┤ │ │ └─────────────────────┘ │ │ │ SecondActivity │ │ │ │ │ ├─────────────────────┤ │ │ GetAbilityRecord() │ │ │ MainActivity ←─┼─── 栈底 (bottom) │ → WeatherAbility │ │ └─────────────────────┘ │ │ │ │ │ │ GetAbilityStackSize() │ │ GetAbilityRecord() │ │ → 1 │ │ → ThirdActivity │ └─────────────────────────┘ │ │ │ GetAbilityStackSize() │ 最近任务列表显示: │ → 3 │ WeatherAbility (1条) └─────────────────────────┘ 最近任务列表显示: ThirdActivity (1条)

3.2 架构分层

┌──────────────────────────────────────────────────────────────────┐ │ Android Framework │ │ IActivityTaskManager.startActivity() │ │ IActivityTaskManager.getRecentTasks() │ │ IActivityTaskManager.removeTask() │ └────────────────────────┬─────────────────────────────────────────┘ │ JNI ┌────────────────────────▼─────────────────────────────────────────┐ │ Adapter Layer │ │ │ │ ActivityTaskManagerAdapter │ │ startActivity(): │ │ → 判断是否应复用现有 Mission (同 bundleName 已有 Mission) │ │ → 是: 调用 OH StartAbilityInMission(missionId, want) │ │ → 否: 调用 OH StartAbility(want) (创建新 Mission) │ │ │ │ getRecentTasks(): │ │ → 直接调用 OH GetMissionInfos() │ │ → 每个 Mission 已经只显示栈顶 Ability，无需额外过滤 │ │ │ │ removeTask(taskId): │ │ → 直接映射 OH CleanMission(missionId) │ │ → Mission 清理时自动销毁栈内所有 Ability │ └────────────────────────┬─────────────────────────────────────────┘ │ OH IPC ┌────────────────────────▼─────────────────────────────────────────┐ │ OH AbilityManagerService │ │ │ │ Mission (修改): │ │ abilityRecord_ → abilityStack_ (vector) │ │ GetAbilityRecord() → 返回栈顶 │ │ PushAbility(record) → 入栈 │ │ PopAbility() → 弹出栈顶 │ │ GetAbilityStackSize() → 栈深度 │ │ │ │ MissionListManager (修改): │ │ StartAbilityInMission() → 新入口，将 Ability 推入已有 Mission │ │ TerminateAbilityLocked() → 弹栈逻辑，栈空时销毁 Mission │ │ │ │ IAbilityManager (修改): │ │ StartAbilityInMission(want, missionId) → 新增 1 个接口方法 │ │ │ │ 其他组件（不修改）: │ │ MissionInfo, InnerMissionInfo, MissionInfoMgr, IMissionManager │ │ MissionList, OH Launcher │ └──────────────────────────────────────────────────────────────────┘

3.3 设计原则

封闭修改原则：所有核心变更封闭在 Mission 类内部。
  • Mission::GetAbilityRecord() 返回栈顶 — 所有调用方（MissionListManager、MissionInfoMgr、MissionSnapshot）无需修改，自动获取栈顶 Ability 的信息
  • MissionInfo 的 label/icon/want 来源于 mission->GetAbilityRecord()，栈顶变化时自然更新
  • GetMissionInfos() 每个 Mission 返回一条 — Android 应用一个 Mission 自然只显示一条
  • OH Launcher 调用 GetMissionInfos() 和 GetMissionSnapshot() — 行为完全兼容

4.2 Mission 方法实现

文件：ability_rt/services/abilitymgr/src/mission/mission.cpp

// ==================== 构造函数（兼容修改）====================

Mission::Mission(int32_t id, MissionAbilityRecordPtr abilityRecord,
                 const std::string &missionName, int32_t startMethod)
    : missionId_(id), startMethod_(startMethod), missionName_(missionName)
{
    // 修改前: abilityRecord_ = abilityRecord;
    // 修改后: 初始 Ability 作为栈底（也是栈顶）
    if (abilityRecord) {
        abilityStack_.push_back(abilityRecord);
    }
}

// ==================== 核心访问方法（行为兼容）====================

MissionAbilityRecordPtr Mission::GetAbilityRecord() const
{
    // 修改前: return abilityRecord_;
    // 修改后: 返回栈顶
    if (abilityStack_.empty()) return nullptr;
    return abilityStack_.back();
}

bool Mission::IsSingletonAbility() const
{
    auto record = GetAbilityRecord();
    return record && record->GetAbilityInfo().launchMode ==
           AppExecFwk::LaunchMode::SINGLETON;
}

bool Mission::IsSpecifiedAbility() const
{
    // 使用栈底 Ability 判断（Mission 的 launch mode 由首个 Ability 决定）
    if (abilityStack_.empty()) return false;
    return abilityStack_.front()->GetAbilityInfo().launchMode ==
           AppExecFwk::LaunchMode::SPECIFIED;
}

bool Mission::IsStandardAbility() const
{
    if (abilityStack_.empty()) return false;
    return abilityStack_.front()->GetAbilityInfo().launchMode ==
           AppExecFwk::LaunchMode::STANDARD;
}

// ==================== 新增栈操作方法 ====================

void Mission::PushAbility(MissionAbilityRecordPtr abilityRecord)
{
    if (!abilityRecord) return;
    abilityRecord->SetMissionId(missionId_);
    abilityStack_.push_back(abilityRecord);

    TAG_LOGI(AAFwkTag::ABILITYMGR,
        "Mission %{public}d: pushed ability, stack size=%{public}zu",
        missionId_, abilityStack_.size());
}

MissionAbilityRecordPtr Mission::PopAbility()
{
    // 不允许弹出最后一个 Ability（Mission 至少需要一个）
    // 最后一个 Ability 的移除走 Mission 销毁流程
    if (abilityStack_.size() <= 1) {
        return nullptr;
    }

    auto top = abilityStack_.back();
    abilityStack_.pop_back();

    TAG_LOGI(AAFwkTag::ABILITYMGR,
        "Mission %{public}d: popped ability, stack size=%{public}zu",
        missionId_, abilityStack_.size());

    return top;
}

size_t Mission::GetAbilityStackSize() const
{
    return abilityStack_.size();
}

MissionAbilityRecordPtr Mission::GetBaseAbilityRecord() const
{
    if (abilityStack_.empty()) return nullptr;
    return abilityStack_.front();
}

MissionAbilityRecordPtr Mission::FindAbilityByName(
    const std::string &abilityName) const
{
    for (auto it = abilityStack_.rbegin(); it != abilityStack_.rend(); ++it) {
        if ((*it)->GetAbilityInfo().name == abilityName) {
            return *it;
        }
    }
    return nullptr;
}

std::vector<MissionAbilityRecordPtr> Mission::PopAbilitiesAbove(
    const std::string &abilityName, bool includeSelf)
{
    std::vector<MissionAbilityRecordPtr> popped;

    // Find the target ability in stack
    auto targetIt = abilityStack_.end();
    for (auto it = abilityStack_.begin(); it != abilityStack_.end(); ++it) {
        if ((*it)->GetAbilityInfo().name == abilityName) {
            targetIt = it;
            break;
        }
    }

    if (targetIt == abilityStack_.end()) return popped;

    // Pop everything above target
    auto startPop = includeSelf ? targetIt : targetIt + 1;
    for (auto it = startPop; it != abilityStack_.end(); ++it) {
        popped.push_back(*it);
    }
    abilityStack_.erase(startPop, abilityStack_.end());

    return popped;
}

4.3 兼容性分析：GetAbilityRecord() 的调用方

GetAbilityRecord() 返回栈顶后，以下所有调用方无需修改，自动获得正确的栈顶 Ability 信息：

调用方使用方式兼容性
MissionListManager::MoveMissionToFront mission->GetAbilityRecord()->ProcessForegroundAbility() 自动将栈顶 Ability 前台化
MissionInfoMgr::BuildInnerMissionInfo targetRecord->GetLabel() → MissionInfo.label label 自动使用栈顶 Ability 的标签
MissionInfoMgr::GetMissionSnapshot missionInfo.want.GetElement() → topAbility want 来自栈顶 Ability，topAbility 自动正确
MissionListManager::TerminateAbilityLocked 查找 ability 所在 mission 需要修改：改为栈式弹出逻辑（见 §5）
MissionList::RemoveMissionByAbilityRecord 按 AbilityRecord 指针查找 Mission 需要修改：在栈中搜索而非直接比较单个指针（见 §5）
Mission::Dump abilityRecord_->Dump() 自动 dump 栈顶；可增强为 dump 全栈
Mission::IsSingletonAbility 判断 launchMode 使用栈顶/栈底判断，行为正确

5. MissionListManager 修改

5.1 新增 StartAbilityInMission 入口

文件：ability_rt/services/abilitymgr/src/mission/mission_list_manager.cpp

现有 StartAbility() 流程始终创建新 Mission。新增 StartAbilityInMission()， 将新 Ability 推入已有 Mission 的栈中。

/**
 * 在已有 Mission 中启动新 Ability（推入栈顶）。
 *
 * 与 StartAbility() 的区别：
 *   - StartAbility(): 创建新 Mission + 新 Ability
 *   - StartAbilityInMission(): 在已有 Mission 内创建新 Ability 并入栈
 *
 * 仅被 Android 适配层调用。OH 原生应用走 StartAbility()。
 *
 * @param abilityRequest  Ability 启动请求（包含 Want、AbilityInfo 等）
 * @param missionId       目标 Mission ID
 * @return ERR_OK on success
 */
int MissionListManager::StartAbilityInMission(
    AbilityRequest &abilityRequest, int32_t missionId)
{
    std::lock_guard<std::recursive_mutex> guard(managerLock_);

    TAG_LOGI(AAFwkTag::ABILITYMGR,
        "StartAbilityInMission: missionId=%{public}d, ability=%{public}s",
        missionId, abilityRequest.abilityInfo.name.c_str());

    // 1. 查找目标 Mission
    std::shared_ptr<Mission> targetMission = nullptr;
    bool isReachToLimit = false;
    auto targetList = GetTargetMissionList(missionId, targetMission, isReachToLimit);

    if (!targetMission) {
        TAG_LOGE(AAFwkTag::ABILITYMGR,
            "StartAbilityInMission: Mission %{public}d not found", missionId);
        return ERR_INVALID_VALUE;
    }

    // 2. 创建新的 AbilityRecord
    auto newRecord = MissionAbilityRecord::CreateAbilityRecord(abilityRequest);
    if (!newRecord) {
        TAG_LOGE(AAFwkTag::ABILITYMGR, "Failed to create AbilityRecord");
        return ERR_INVALID_VALUE;
    }

    // 3. 将当前栈顶 Ability 调度到后台
    auto currentTop = targetMission->GetAbilityRecord();
    if (currentTop && currentTop->GetAbilityState() == AbilityState::FOREGROUND) {
        currentTop->ProcessBackgroundAbility();
    }

    // 4. 推入新 Ability 到 Mission 栈
    targetMission->PushAbility(newRecord);

    // 5. 将 Mission 移到 MissionList 顶部
    targetList->AddMissionToTop(targetMission);

    // 6. 调度新 Ability 前台化
    newRecord->ProcessForegroundAbility(...);

    // 7. 更新 MissionInfo（label/want 自动更新为栈顶）
    UpdateMissionInfo(targetMission);

    return ERR_OK;
}

5.2 修改 TerminateAbilityLocked — 栈式弹出

文件：ability_rt/services/abilitymgr/src/mission/mission_list_manager.cpp

现有逻辑：Ability 终止时从 MissionList 中移除对应 Mission。
修改后：如果 Mission 有多个 Ability（多 Ability 模式），弹出栈顶而非移除整个 Mission。

int MissionListManager::TerminateAbilityLocked(
    const std::shared_ptr<AbilityRecord> &abilityRecord, int flag)
{
    auto missionId = abilityRecord->GetMissionId();
    auto mission = GetMissionById(missionId);
    if (!mission) {
        return ERR_INVALID_VALUE;
    }

    // ========== 新增逻辑：多 Ability 栈模式 ==========
    if (mission->IsMultiAbilityMode() && mission->GetAbilityStackSize() > 1) {
        // 弹出栈顶 Ability
        auto popped = mission->PopAbility();

        if (popped) {
            // 添加到终止队列
            terminateAbilityList_.push_back(popped);

            // 调度弹出后的新栈顶 Ability 到前台
            auto newTop = mission->GetAbilityRecord();
            if (newTop) {
                newTop->ProcessForegroundAbility(...);
            }

            // 更新 Mission 的 MissionInfo（label/want 变为新栈顶）
            UpdateMissionInfo(mission);

            // 通知 MissionListener 更新（可选，用于 Launcher 刷新快照）
            listenerController_->NotifyMissionSnapshotChanged(missionId);

            TAG_LOGI(AAFwkTag::ABILITYMGR,
                "TerminateAbility: popped from stack, mission %{public}d "
                "remaining=%{public}zu", missionId, mission->GetAbilityStackSize());
        }

        return ERR_OK;
    }

    // ========== 现有逻辑（OH 原生应用，栈内只有 1 个 Ability）==========
    // 原有的 RemoveTerminatingAbility + Terminate 流程完全不变
    RemoveTerminatingAbility(abilityRecord, flag);
    abilityRecord->SendResultToCallers();
    // ... (现有代码不修改) ...

    return ERR_OK;
}

5.3 修改 MissionList::RemoveMissionByAbilityRecord

文件：ability_rt/services/abilitymgr/src/mission/mission_list.cpp

现有逻辑：按 mission->GetAbilityRecord() == abilityRecord 精确匹配。
修改后：如果是多 Ability 模式，在栈中搜索。

std::shared_ptr<Mission> MissionList::GetMissionByAbilityRecord(
    const std::shared_ptr<AbilityRecord> &abilityRecord) const
{
    for (auto &mission : missions_) {
        if (mission->IsMultiAbilityMode()) {
            // 多 Ability 模式：在栈中搜索
            if (mission->FindAbilityByName(abilityRecord->GetAbilityInfo().name)) {
                return mission;
            }
        } else {
            // 单 Ability 模式：原有比较逻辑
            if (mission->GetAbilityRecord() == abilityRecord) {
                return mission;
            }
        }
    }
    return nullptr;
}

5.4 新增 UpdateMissionInfo 辅助方法

/**
 * 更新 Mission 在 MissionInfoMgr 中的信息。
 * 当栈顶 Ability 变化时（push/pop），需要更新 label、want 等。
 */
void MissionListManager::UpdateMissionInfo(const std::shared_ptr<Mission> &mission)
{
    auto topRecord = mission->GetAbilityRecord();
    if (!topRecord) return;

    InnerMissionInfo info;
    if (DelayedSingleton<MissionInfoMgr>::GetInstance()
            ->GetInnerMissionInfoById(mission->GetMissionId(), info) != ERR_OK) {
        return;
    }

    // 更新为栈顶 Ability 的信息
    info.missionInfo.want = topRecord->GetWant();
    info.missionInfo.label = topRecord->GetLabel();
    info.missionInfo.time = GetCurrentTime();

    DelayedSingleton<MissionInfoMgr>::GetInstance()
        ->UpdateMissionInfo(info);
}

6.2 AbilityManagerService 实现

int AbilityManagerService::StartAbilityInMission(
    const Want &want, int32_t missionId)
{
    TAG_LOGI(AAFwkTag::ABILITYMGR,
        "StartAbilityInMission: mission=%{public}d, ability=%{public}s",
        missionId, want.GetElement().GetAbilityName().c_str());

    // Build AbilityRequest from Want
    AbilityRequest abilityRequest;
    int result = GenerateAbilityRequest(want, -1, abilityRequest, nullptr, 0);
    if (result != ERR_OK) {
        return result;
    }

    // Delegate to MissionListManager
    auto userId = GetUserId();
    auto missionListMgr = GetMissionListManagerByUserId(userId);
    if (!missionListMgr) {
        return ERR_INVALID_VALUE;
    }

    return missionListMgr->StartAbilityInMission(abilityRequest, missionId);
}

7.2.2 startActivity 适配实现

// ActivityTaskManagerAdapter.java

// 跟踪 Android bundleName → OH missionId 的映射
private final Map<String, Integer> mActiveMissions = new HashMap<>();

@Override
public int startActivity(IApplicationThread caller, String callingPackage,
        String callingFeatureId, Intent intent, String resolvedType,
        IBinder resultTo, String resultWho, int requestCode,
        int flags, ProfilerInfo profilerInfo, Bundle options) throws RemoteException {
    logBridged("startActivity", "-> OH StartAbility / StartAbilityInMission");

    String bundleName = ...; // 从 intent.getComponent() 提取
    String abilityName = ...;

    // 判断是否需要创建新 Task
    boolean forceNewTask = (flags & Intent.FLAG_ACTIVITY_NEW_TASK) != 0
                        && (flags & Intent.FLAG_ACTIVITY_MULTIPLE_TASK) != 0;

    if (!forceNewTask && mActiveMissions.containsKey(bundleName)) {
        // 复用已有 Mission：将新 Activity 推入 Ability 栈
        int missionId = mActiveMissions.get(bundleName);

        // 处理 FLAG_ACTIVITY_CLEAR_TOP
        if ((flags & Intent.FLAG_ACTIVITY_CLEAR_TOP) != 0) {
            nativeClearAbilitiesAbove(missionId, abilityName);
        }

        // 处理 FLAG_ACTIVITY_SINGLE_TOP
        if ((flags & Intent.FLAG_ACTIVITY_SINGLE_TOP) != 0) {
            if (nativeIsTopAbility(missionId, abilityName)) {
                // 栈顶已是目标 Ability，发送 onNewIntent 而不创建新实例
                nativeDeliverNewIntent(missionId, abilityName, intent);
                return 0;
            }
        }

        int result = nativeStartAbilityInMission(bundleName, abilityName,
                                                  missionId, ...);
        return result;
    } else {
        // 创建新 Mission
        int missionId = nativeStartAbility(bundleName, abilityName, ...);
        if (missionId >= 0) {
            // 标记 Mission 为多 Ability 模式
            nativeSetMultiAbilityMode(missionId, true);
            mActiveMissions.put(bundleName, missionId);
        }
        return missionId >= 0 ? 0 : missionId;
    }
}

7.2.3 getRecentTasks 适配实现

@Override
public ParceledListSlice<ActivityManager.RecentTaskInfo> getRecentTasks(
        int maxNum, int flags, int userId) throws RemoteException {
    logBridged("getRecentTasks", "-> OH GetMissionInfos (栈顶自动正确)");

    // 直接调用 OH GetMissionInfos — 无需额外分组或过滤！
    // 因为 Android 应用的多个 Activity 已在同一 Mission 内，
    // 每个 Mission 自然只返回一条，且信息来自栈顶 Ability。
    MissionInfoData[] missions = nativeGetMissionInfos(maxNum);

    List<ActivityManager.RecentTaskInfo> result = new ArrayList<>();
    for (MissionInfoData mission : missions) {
        ActivityManager.RecentTaskInfo taskInfo = new ActivityManager.RecentTaskInfo();
        taskInfo.taskId = mission.id;
        taskInfo.persistentId = mission.id;
        taskInfo.topActivity = new ComponentName(
            mission.bundleName, mission.topAbilityName);
        // numActivities = Mission 的 AbilityStackSize
        taskInfo.numActivities = mission.stackSize;
        taskInfo.isRunning = mission.runningState >= 0;
        taskInfo.lastActiveTime = mission.lastActiveTime;
        taskInfo.taskDescription = new ActivityManager.TaskDescription(
            mission.label, 0, 0);
        taskInfo.baseIntent = new Intent();
        taskInfo.baseIntent.setComponent(taskInfo.topActivity);
        result.add(taskInfo);
    }

    return new ParceledListSlice<>(result);
}

7.2.4 removeTask 适配实现

@Override
public boolean removeTask(int taskId) throws RemoteException {
    logBridged("removeTask", "-> OH CleanMission");
    // taskId 直接映射为 OH missionId
    // CleanMission 会终止栈内所有 Ability 并销毁 Mission
    int result = nativeCleanMission(taskId);

    // 清理本地映射
    mActiveMissions.values().removeIf(id -> id == taskId);

    return result == 0;
}

9. Mission 清理（CleanMission）

9.1 多 Ability Mission 的清理流程

当用户在最近任务列表中划掉一个 Android 应用的 Mission 时：

// MissionListManager::CleanMission 修改
int MissionListManager::CleanMission(int32_t missionId)
{
    auto mission = GetMissionById(missionId);
    if (!mission) return ERR_INVALID_VALUE;

    // 如果是多 Ability 模式，需要终止栈内所有 Ability
    if (mission->IsMultiAbilityMode()) {
        // 从栈顶到栈底逐个终止
        while (mission->GetAbilityStackSize() > 1) {
            auto popped = mission->PopAbility();
            if (popped) {
                popped->Terminate(/* cleanup */);
                terminateAbilityList_.push_back(popped);
            }
        }
    }

    // 最后一个 Ability 走现有清理逻辑
    // (移除 Mission from MissionList, DeleteMissionInfo, 等)
    return CleanMissionLocked(missionId, mission);
}

10. Intent Flags 处理

Android Intent Flag在 Mission Ability 栈中的行为
FLAG_ACTIVITY_NEW_TASK
(无 MULTIPLE_TASK)		 查找该 bundleName 的活跃 Mission：
→ 存在：复用 Mission，新 Ability 入栈
→ 不存在：创建新 Mission
等同于默认行为（Adapter 层已实现此逻辑）
FLAG_ACTIVITY_NEW_TASK
| FLAG_ACTIVITY_MULTIPLE_TASK		 强制创建新 Mission（不复用已有 Mission）
用于多窗口/文档模式
FLAG_ACTIVITY_CLEAR_TOP 在栈中查找目标 Ability：
→ 找到：PopAbilitiesAbove(abilityName) 弹出其上方所有 Ability
→ 未找到：正常入栈
FLAG_ACTIVITY_SINGLE_TOP 检查栈顶 Ability 是否为目标：
→ 是：不创建新实例，调用 onNewIntent()
→ 否：正常入栈
FLAG_ACTIVITY_CLEAR_TASK 清空整个栈，然后新 Ability 作为唯一成员
等同于重新创建 Mission
FLAG_ACTIVITY_REORDER_TO_FRONT 在栈中查找目标 Ability，将其移到栈顶
不新建实例，仅调整栈序

12.2 Ability 栈的持久化

Ability 栈的持久化通过在现有 JSON 中增加可选字段实现：

// mission_5.json (现有格式，向后兼容)
{
    "missionId": 5,
    "bundleName": "com.example.app",
    "missionName": "com.example.app#MainActivity",
    "launchMode": 0,
    "missionInfo": {
        "id": 5,
        "label": "ThirdActivity",        // 始终为栈顶 Ability 的 label
        "want": { ... },                  // 始终为栈顶 Ability 的 want
        "time": "2026-03-11 10:30:00"
    },

    // ========== 新增可选字段 ==========
    "multiAbilityMode": true,
    "abilityStack": [
        // 栈底 → 栈顶
        { "abilityName": "MainActivity", "want": { ... } },
        { "abilityName": "SecondActivity", "want": { ... } },
        { "abilityName": "ThirdActivity", "want": { ... } }
    ]
}
向后兼容：
  • multiAbilityMode 和 abilityStack 为可选字段
  • OH 原生应用的 JSON 文件没有这些字段，现有解析逻辑不受影响
  • 系统重启后，如果存在 abilityStack，则恢复整个栈；否则按现有逻辑恢复单个 Ability

15. 风险与缓解

风险影响缓解措施
Mission 其他方法假设只有一个 Ability 调用 GetAbilityRecord() 后可能有隐含假设 全面搜索 GetAbilityRecord() 的调用点，逐个验证兼容性。
核心保证：GetAbilityRecord() 返回栈顶，语义从 "唯一" 变为 "当前活跃"，大多数场景兼容。
OH 版本升级后 Mission 类变更 栈化修改需要重新适配 修改范围仅限 Mission 内部，且保持 GetAbilityRecord() 签名不变。OH 升级时只需 rebase mission.h/cpp。
多 Ability 模式下内存占用增加 Android 应用的 Activity 栈可能较深 设置栈深度上限（如 20），超限时弹出最底部的 Ability；与 Android 的 Activity 数量限制一致。
Ability 生命周期回调顺序 栈内非顶 Ability 处于后台，需正确处理 onStop/onStart Push 时对当前栈顶调用 ProcessBackgroundAbility()；Pop 时对新栈顶调用 ProcessForegroundAbility()。
