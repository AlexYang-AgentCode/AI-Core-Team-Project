1. 概述

本文档描述从OpenHarmony系统服务到Android应用的反向回调桥接层设计。与正向桥接（App调用系统服务）相反，反向桥接处理的是系统服务主动通知应用的场景，例如：

  • 系统通知应用进入前台/后台（生命周期回调）
  • 窗口管理器通知窗口大小变化、焦点变化
  • 服务连接建立/断开通知
  • 内存不足警告
  • 配置变更通知

设计目标

  • 完整覆盖：OH侧6个回调接口的每个方法都有明确的处理策略
  • 影响评估：无法映射的方法标注对App正常工作的影响等级（高/中/低/无）
  • 处理策略：给出每个无法映射方法的处理建议（忽略、修改OH服务、增强能力等）
  • Android兼容：确保Android应用能正确接收必要的系统回调

2.2 回调方向与数据流

反向回调的完整数据流（C++ Stub继承架构）：

  1. OH系统服务通过IPC调用应用进程中的OH回调接口（IAppScheduler等）
  2. C++ Stub适配器（继承OH Stub类，如AppSchedulerAdapter : public AppSchedulerHost）直接接收OH IPC调用，每个override方法被触发
  3. JNI调用：C++适配器通过JavaVM->AttachCurrentThread()获取JNIEnv，将OH参数转换后调用Java层方法
  4. Java桥接层（Bridge类）将OH回调参数最终转换为Android格式
  5. Android框架通过IApplicationThread/IWindow/IServiceConnection将回调传递给Android应用代码

关键设计：C++适配器直接继承OH的IPC Stub类，作为真正的OH IPC端点参与Binder通信，无需中间的回调转发层。OH系统服务持有适配器的sptr<IRemoteObject>引用，直接进行IPC调用。

2.3 桥接模式

反向桥接采用C++ Stub类继承模式，与正向桥接（Java extends IXxx.Stub）对称：

C++层：继承OH Stub类

// C++ 适配器 - 直接继承OH IPC Stub类
class AppSchedulerAdapter : public OHOS::AppExecFwk::AppSchedulerHost {
private:
    JavaVM* jvm_;
    jobject mApplicationThread_;  // Java IApplicationThread.Stub 全局引用

public:
    AppSchedulerAdapter(JavaVM* vm, jobject appThread);

    // OH系统服务通过IPC调用此方法
    void ScheduleLaunchAbility(const AbilityInfo& info,
                               const sptr<IRemoteObject>& token,
                               const shared_ptr<Want>& want,
                               int32_t recordId) override {
        JNIEnv* env = getJNIEnv();
        // 通过JNI调用Java层方法
        env->CallVoidMethod(mApplicationThread_,
            scheduleTransactionMethod, ...);
    }
};

Java层：参数转换与Android接口调用

// Java 桥接类 - 由C++适配器通过JNI调用
public class AppSchedulerBridge {
    private final Object mApplicationThread; // Android IApplicationThread

    // 从C++层通过JNI调用
    public void onScheduleForegroundApplication() {
        // 转换参数并调用Android回调接口
        invokeApplicationThread("setProcessState",
            new Class[]{int.class}, new Object[]{PROCESS_STATE_TOP});
    }
}

6.1 ServiceConnectionRegistry — 连接注册表

源文件：D:\code\adapter\framework\java\adapter\client\ServiceConnectionRegistry.java

设计概述

ServiceConnectionRegistry 是一个单例注册表，管理 Android IServiceConnection 与 OH AbilityConnection 之间的映射关系。当 Android 应用调用 bindService() 时，适配层将 IServiceConnection 注册到此注册表并获得一个本地 connectionId；该 id 传递给 native 侧创建 OH AbilityConnection 后，OH 返回的 ohConnectionId 通过 setOhConnectionId() 与本地 id 关联。当 OH 触发 OnAbilityConnectDone / OnAbilityDisconnectDone 回调时，native 侧通过此注册表查找对应的 Android IServiceConnection 并分发事件。

核心数据结构

  • ConnectionRecord：内部记录类，持有 IServiceConnection、目标 ComponentName、本地 id、OH id、绑定状态
  • mConnections：Map<IBinder, ConnectionRecord> — 以 IServiceConnection.asBinder() 为键，用于注册/注销查找
  • mConnectionsByOhId：Map<Integer, ConnectionRecord> — 以 OH connectionId 为键，用于回调分发
  • mConnectionsByLocalId：Map<Integer, ConnectionRecord> — 以本地 connectionId 为键，用于 setOhConnectionId() 关联

主要方法

方法调用方说明
registerConnection(conn, target) ActivityManagerAdapter.bindService 注册连接，分配本地 id，去重检查
unregisterConnection(conn) ActivityManagerAdapter.unbindService 注销连接，返回 ohConnectionId 供 native 断开
setOhConnectionId(localId, ohId) JNI 回调 将本地 id 与 OH 分配的 id 关联
onServiceConnected(ohId, bundle, ability, binder) JNI 回调（OH OnAbilityConnectDone） 查找记录并调用 IServiceConnection.connected(name, binder, false)
onServiceDisconnected(ohId, bundle, ability) JNI 回调（OH OnAbilityDisconnectDone） 查找记录并调用 IServiceConnection.connected(name, null, true)
disconnectAll() OHEnvironment.shutdown() 清除所有映射，关闭时调用

线程安全

所有公共方法均使用 synchronized 关键字保证线程安全。注册操作来自应用主线程（bindService），而回调分发来自 native OH IPC 线程，因此同步是必要的。

7. IWindow(OH) 桥接

10. 影响评估与处理策略

10.1 无法转换到Android接口的OH回调

高影响 (HIGH)

OH回调影响描述处理策略
ISessionStage.HandleBackEvent 返回导航是Android应用的核心交互。如果返回键不工作，用户无法正常退出页面。 注入按键事件：通过InputChannel注入KEYCODE_BACK按键事件。这是最可靠的方案，可以触发Activity.onBackPressed和Fragment.onBackPressedDispatcher。

中等影响 (MEDIUM)

OH回调影响描述处理策略
UpdateFocusStatus / UpdateFocus 焦点变化影响键盘弹出/隐藏、动画触发、UI状态更新 内部路由：通过反射调用ViewRootImpl.windowFocusChanged触发Android焦点变化流程
IAbilityScheduler数据操作（Insert/Update/Delete/Query等） OH DataAbility操作无法直接映射到Android ContentProvider，数据格式不同 格式转换：实现OH NativeRdb.ValuesBucket ↔ ContentValues 转换器，OH DataAbilityPredicates ↔ SQL selection 转换器
SchedulePrepareTerminateAbility OH允许Ability在终止前做准备，Android没有此机制 模拟：在DestroyActivityItem之前先发送PauseActivityItem + StopActivityItem，给App清理时间
PiP相关回调 画中画控制对媒体应用很重要 转发：通过Activity.onPictureInPictureModeChanged和PictureInPictureParams机制实现
NotifyAccessibilityWindowInfo 无障碍服务需要窗口信息 增强OH服务：需要OH WMS提供与Android兼容的无障碍窗口信息格式，或实现格式转换
ScheduleAcceptWant OH多实例场景（specified进程创建） 近似映射：映射到Service的rebind场景，或忽略（大部分App不使用多实例）

低影响 / 无影响

以下OH特有回调对Android应用正常工作无影响，建议忽略：

  • ArkTS/CJ引擎相关：ScheduleJsHeapMemory, ScheduleCjHeapMemory, ScheduleChangeAppGcState — Android使用ART虚拟机，有自己的GC管理
  • 热修复/热重载：ScheduleNotifyLoadRepairPatch, ScheduleNotifyHotReloadPage — Android有自己的热更新机制（InstantRun/Apply Changes）
  • 分布式流转：ContinueAbility, NotifyContinuationResult — Android无内置设备流转
  • IPC诊断：ScheduleDumpIpcStart/Stop/Stat, ScheduleDumpFfrt — 仅诊断功能
  • OH UI特有功能：UpdateTitleInTargetPos, NotifySingleHandTransformChange, SwitchFreeMultiWindow, SetSplitButtonVisible等 — OH桌面/窗口管理特有
  • OH扩展机制：SendExtensionData, NotifyTransferComponentData — OH UIExtension特有
  • 窗口管理器状态：UpdateCameraFloatWindowStatus, NotifyWaterMarkFlagChangedResult, NotifyWindowStyleChange等 — 系统级UI，非应用关注
