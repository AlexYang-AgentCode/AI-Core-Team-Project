1. Project Overview

1.1 Background and Objectives

The goal of this project is to run unmodified Android applications on an OpenHarmony (OH) system. The core approach is to build an Adapter Layer between the Android Framework and OH system services that intercepts Android App system service calls issued via Binder IPC, converts them, and routes them to equivalent OH system service interfaces.

This approach differs from a full Android container or virtual machine solution: we retain the Android Runtime (ART) and Framework layer, but replace the underlying system services with OH implementations. Android Apps run transparently on top of the OH kernel and system services without any awareness of the change.

Target Hardware: HH-SCDAYU600 (DAYU600) development board, equipped with Unisoc UIS7885 (ums9620) SoC.

1.2 Core Design Principles

Principle 1: App-to-System-Service Interface Bridging

The core responsibility of the adapter layer is to achieve bidirectional transparent communication between Android Apps and OH system services. In the forward direction, it intercepts system service calls issued by Android Apps via Binder IPC (such as ActivityManager, WindowManager, PackageManager) and converts them through adapters into calls to equivalent OH system services (such as AbilityManager, SceneSessionManager, BundleManager), allowing Apps to transparently use OH system capabilities without any modification. In the reverse direction, callbacks actively pushed by OH system services (such as IAbilityScheduler lifecycle notifications, ISessionStage window events, IAbilityConnection service connection notifications) pass through the callback bridge layer and are converted into standard callback interfaces recognizable by Android Apps (such as IApplicationThread, IWindow, IServiceConnection), ensuring that App lifecycle management, window interaction, and service binding mechanisms work correctly. The entire bridging process is completely transparent to the App, with all 530 forward methods and 210 reverse callback methods covered.

Principle 2: Framework Self-Rendering

The complete Android Framework rendering pipeline is retained; the App's UI rendering is entirely handled by the Android Framework itself, and the adapter layer does not intervene in the rendering process. Android's View system (Measure → Layout → Draw), RenderThread hardware-accelerated rendering, Skia 2D graphics library, HWUI hardware abstraction layer, and other components run unmodified within ART. The rendered GraphicBuffers are submitted to OH RenderService for composition and display through the Surface bridge module (OHGraphicBufferProducer). Both Android and OH graphics buffers are based on the Linux dmabuf mechanism, and the BufferQueue state machine can be mapped 1:1 (FREE↔RELEASED, DEQUEUED↔REQUESTED, QUEUED↔FLUSHED, ACQUIRED↔ACQUIRED), ensuring the feasibility of cross-runtime buffer sharing. This design avoids the enormous engineering effort of reimplementing the rendering pipeline while preserving the native rendering quality and performance of Android Apps.

Principle 3: Class Inheritance Bridging

All forward adapters use class inheritance (extends IXxx.Stub) rather than dynamic proxy (Dynamic Proxy). Each adapter directly inherits the AIDL-generated Stub abstract class, explicitly overriding every interface method, gaining complete compile-time type checking and method signature verification. Compared to the dynamic proxy approach (intercepting all calls through InvocationHandler and dispatching by method name string matching), class inheritance eliminates runtime reflection overhead, supports IDE code navigation and safe refactoring, and when interfaces change (e.g., AOSP version upgrades) the compiler reports errors immediately, avoiding compatibility issues that would only surface at runtime. Each adapter declares its own native methods for OH service access and holds independent OH service handles (e.g., long mOhAbilityManager), maintaining clear boundaries between modules.

Principle 4: App Spawning Process — Conform Downward to OH, Contain ART Upward

Android App process spawning is handled by the appspawn-x hybrid spawner, designed following the "conform downward, contain upward" principle. Downward, appspawn-x fully conforms to OH's process management mechanism: it is scheduled uniformly by OH foundation, receives spawn requests via OH's standard socket protocol, and the forked child process strictly follows OH security specialization procedures (UID/GID setup, SELinux context switching, AccessToken binding, sandbox mounting), ensuring that Android App processes are entirely consistent with OH native processes in terms of security model and process control, without introducing any security model exceptions or bypasses. Upward, appspawn-x integrates a complete ART (Android Runtime) virtual machine within the process, supporting Dex bytecode loading and execution. ART provides JIT/AOT compilation, garbage collection, JNI calling, and other core capabilities; classes.dex in APKs are pre-compiled to native machine code (.odex/.vdex) via dex2oat for runtime performance. appspawn-x pre-loads the ART VM and approximately 7,000 Framework core classes and resources before fork, using COW (Copy-On-Write) to enable memory sharing across multiple processes, significantly reducing App startup latency and memory consumption. After fork, the child process enters ActivityThread.main() → Looper.loop(), running the standard Android application main loop within ART.

Principle 5: Java-C++ Dual-Language Adaptation

The adapter layer is implemented through Java and C++ dual-language collaboration, connected via JNI (Java Native Interface). This choice is dictated by the technology stacks of both systems: Android's AIDL interfaces, Framework data types (Intent, Bundle, Parcelable, etc.), and component mechanisms (ClientTransaction, ActivityRecord, etc.) are all Java implementations, so the adapter's Stub inheritance and Android-side interactions must use Java; while OH system service IPC interfaces are based on C++ IRemoteProxy/IRemoteBroker mechanisms, so the adapter's communication with OH services must use C++. The two languages are connected through the liboh_adapter_bridge.so shared library — in forward calls, Java adapters invoke C++ layer OH IPC through native methods; in reverse callbacks, C++ Stubs obtain JNIEnv via AttachCurrentThread and call Java Bridge methods, with each C++ adapter caching Java object references (jobject) and method IDs (jmethodID) to avoid repeated lookup overhead.

Principle 6: Minimal OH Modification

All compatibility issues are preferably resolved within the adapter layer; OH system source code is modified only when absolutely unavoidable. All OH-side modifications are managed as patch files (stored in the ohos_patches/ directory) for easy tracking, review, and synchronized maintenance across OH version upgrades. The current OH modification scope is limited to two subsystems: ability_rt (ability management framework, 5 files) to support Ability stacks within Missions and Android process management; bundle_framework (package management framework, 4 files) to support APK installation and Android package type recognition. All other adaptation logic (interface conversion, data format mapping, callback bridging, etc.) is self-contained within the adapter layer, and OH Launcher, SystemUI, and other upper-layer applications require no modifications.

2.2 Layered Architecture

The overall adaptation solution consists of the following five layers:

Layer 1: Android Application Layer

Standard Android Apps (Activity, Service, BroadcastReceiver, ContentProvider) using standard Android SDK APIs, running without any modifications.

Layer 2: Android Framework Layer (Modified AOSP 14)

Environment detection logic is inserted at key service acquisition points (ActivityManager, ActivityTaskManager, WindowManagerGlobal, ActivityThread): when OHEnvironment.isOHEnvironment() returns true, adapter instances are returned instead of original Binder proxies. A total of 4 AOSP source files are modified.

Layer 3: Adapter Layer (Core of This Project)

Located in the framework/ directory. Contains forward adapters (5 Stub subclasses), reverse callback bridges (6 Bridge classes + 5 C++ Stub adapters), data converters (Intent↔Want, Broadcast↔CommonEvent, URI mapping, etc.), and the JNI layer (liboh_adapter_bridge.so).

Layer 4: OH System Service Layer

Standard OH system services (AbilityManagerService, AppMgrService, WindowManagerService, RenderService, CommonEventService, DataShareService, BundleManagerService). Extended via patches (ohos_patches/) only when necessary.

Layer 5: Process Management Layer (appspawn-x)

A hybrid spawner combining OH's security model with Android Zygote's ART preloading. Responsible for creating and initializing Android App processes.

3.2 Interface Feasibility Analysis Conclusions

A per-method feasibility analysis was conducted on the 4 major Android system service interfaces (530 forward methods) and 6 callback interfaces (181 reverse methods):

Key Findings

  • 43% directly mappable — Approximately 305 methods across forward/reverse interfaces can be directly or logically mapped to OH interfaces
  • Only 14% need OH enhancement — Of the initially estimated 55% "unmappable" methods, after three-way classification only 102 truly require new capabilities on the OH side
  • 43% can be safely ignored — Multi-user, Instrumentation, VR/voice interaction, lock screen/wallpaper, debug tracing, and other features are not needed in the current target scenario
  • IActivityTaskManager has the highest mapping rate (53%); broadcast and ContentProvider have the lowest mapping rates (requiring dedicated adaptation designs)

4. Forward Bridge Design

4.1 Bridge Pattern: Class Inheritance

All forward adapters use the class inheritance pattern (extends IXxx.Stub) rather than dynamic proxy. This choice is based on the following considerations:

ComparisonClass Inheritance (Adopted)Dynamic Proxy (Rejected)
Type SafetyCompile-time checking of all method signaturesParameter type errors only discovered at runtime
PerformanceDirect method calls, no reflection overheadEvery call goes through InvocationHandler reflection
MaintainabilityIDE navigable, refactor-safeMethod name string matching, error-prone
Code VolumeMore (must explicitly override every method)Less (unified forwarding logic)

Basic structure of each adapter:

public class ActivityManagerAdapter extends IActivityManager.Stub {

4.5 Lifecycle Mapping

LifecycleAdapter handles the mapping between Android's 6 states and OH's 4 states:

Android Lifecycle OH Ability Lifecycle ═══════════════ ════════════════════ CREATED ──────────────→ INITIAL STARTED ──────────────→ INACTIVE RESUMED ──────────────→ FOREGROUND PAUSED ──────────────→ INACTIVE (fallback) STOPPED ──────────────→ BACKGROUND DESTROYED ────────────→ (released, no corresponding state)
Note: OH does not have a state precisely equivalent to Android's PAUSED. PAUSED maps to INACTIVE, but INACTIVE is also the mapping target for STARTED. The adapter layer distinguishes between these two cases through internal state tracking within LifecycleAdapter.

5. Reverse Callback Bridge Design

5.1 Callback Direction and Threading Model

Reverse callbacks are calls actively pushed from OH system services to the App (such as lifecycle transitions, window configuration changes). Since OH uses C++ Binder, callbacks arrive on the OH Binder thread and must travel through the following path to reach the Android App:

OH System Service │ ▼ (OH Binder IPC) C++ Stub Adapter (OH Binder thread) │ ▼ JNI: AttachCurrentThread + CallVoidMethod Java Bridge Class (still on Binder thread) │ ▼ Android IPC (Binder/Handler) Android App Component (main thread)

5.2 Six Callback Adapters

5.3 Two-Layer Architecture: C++ Stub + Java Bridge

The reverse callback adopts a two-layer architecture for the following reasons:

  • C++ Stub Layer: Inherits OH's IRemoteStub subclass, handles OH Binder OnRemoteRequest message parsing. This layer must be implemented in C++ because OH Binder is a C++ native interface.
  • Java Bridge Layer: Holds references to Android IApplicationThread / IWindow / IServiceConnection, responsible for converting OH callback data into Android API calls. This layer is implemented in Java because Android Binder callback interfaces are generated by Java AIDL.
  • JNI Connection: C++ Stubs obtain JNIEnv via AttachCurrentThread and call corresponding Java Bridge methods. Each C++ adapter caches the Java Bridge's jobject and key methods' jmethodID.

6.2 Surface / Graphics Buffer Bridging

Graphics buffer bridging is the most technically challenging part of the entire project. Key discovery:

Key Feasibility Confirmation: OH's BufferHandle.fd is a standard Linux dmabuf file descriptor, using the same kernel mechanism as Android GraphicBuffer. OH's SyncFence also uses Linux sync_file fd. This confirms that cross-runtime buffer sharing is technically feasible.

The OHGraphicBufferProducer adapter wraps an OH Surface as an Android IGraphicBufferProducer interface:

7. Four Major Component Adaptation

7.1 Activity ↔ Ability Adaptation

Activity adaptation is the core scenario, implemented through ActivityManagerAdapter + ActivityTaskManagerAdapter forward bridging, and AbilitySchedulerBridge + AppSchedulerBridge reverse callbacks.

startActivity Complete Call Chain

  1. App calls startActivity(intent)
  2. Instrumentation.execStartActivity() → ATMS.getService().startActivity()
  3. AOSP modification point returns ActivityTaskManagerAdapter
  4. ActivityTaskManagerAdapter.startActivity() internally:
    • IntentWantConverter.intentToWant(intent) → OH Want
    • nativeStartAbility(mOhAbilityManager, want) → JNI
  5. C++ layer calls AbilityManagerService.StartAbility(want) via OH IPC
  6. OH AMS calls back via IAbilityScheduler
  7. AbilitySchedulerAdapter (C++) → JNI → AbilitySchedulerBridge (Java)
  8. LifecycleAdapter.mapOHToAndroid() → ClientTransaction → Activity.onResume()

7.2 Service ↔ ServiceExtensionAbility Adaptation

Service adaptation covers three scenarios:

Scenario S1: Android App → Android Service (Both Running on OH)

The most common scenario. startService(intent) is routed through the adapter layer to OH AbilityManager.StartAbility(), and OH creates the Service in the target process via appspawn-x.

Scenario S2: Android App → OH ServiceExtension

The forward path is the same as S1. Connection callbacks are bridged back to the Android App via IAbilityConnection → AbilityConnectionBridge → IServiceConnection.

Scenario S3: OH App → Android Service (Reverse Bridge)

Triggers the complete Android Service creation flow: scheduleCreateService + scheduleBindService.

ServiceConnectionRegistry tracks the mapping between IServiceConnection ↔ OH IAbilityConnection, handling concurrent connections and automatic cleanup on death listener.

Detailed Document: Service → ServiceExtensionAbility Adaptation Design

8.2 Built-in Ability Stack within Mission

The v2.0 approach is adopted: extending the Mission class internally to support an Ability stack:

  • Extend abilityRecord_ single pointer to an abilityStack_ vector
  • GetAbilityRecord() returns the top-of-stack element, all external interfaces remain unchanged
  • Add 1 new IAbilityManager method: StartAbilityInMission(want, missionId)
  • Closed modification principle: All changes are encapsulated within the Mission class, OH Launcher requires no modifications
  • Only 3 OH system files + 3 Adapter files modified (compared to v1.0's 8+6 files)

13. Implementation Plan

The project follows a phased implementation strategy:

Phase 1: Hello World Verification (Design Completed)

  • Basic framework setup: OHEnvironment, ServiceInterceptor, JNI Bridge
  • ActivityManagerAdapter / ActivityTaskManagerAdapter core methods (startActivity)
  • IntentWantConverter, LifecycleAdapter basic implementation
  • Hello World test App (startActivity call chain verification)
  • Stub mode operation (logging simulated OH IPC calls)

Phase 2: Full Lifecycle + Four Major Components

  • Forward bridge 530 methods fully covered (BRIDGED-tagged methods connected to real OH IPC)
  • Reverse callback 6 major bridge implementations (210 methods)
  • Service ↔ ServiceExtensionAbility three scenarios
  • BroadcastReceiver ↔ CommonEventService
  • ContentProvider ↔ DataShareExtensionAbility
  • Mission Group (Android Task multi-Activity stack)

Phase 3: Graphics and Window System

  • WindowManagerAdapter / WindowSessionAdapter connected to real OH window services
  • OHGraphicBufferProducer — Surface buffer bridging
  • RSSurfaceNode + RSUIDirector management
  • InputEventBridge — MMI to Android InputEvent conversion
  • Graphics composition pipeline verification

Phase 4: Process Management and Package Management

  • appspawn-x hybrid spawner implementation
  • APK installation flow (parse → signature verification → deploy → BMS registration)
  • PackageManagerAdapter connected to OH BMS
  • PermissionMapper — Permission system mapping
  • End-to-end APK installation + runtime verification
