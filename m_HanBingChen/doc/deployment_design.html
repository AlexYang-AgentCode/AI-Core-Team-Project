1.4 部署原则

  1. 不刷机 — 不烧写分区镜像，仅通过 hdc file send 向设备推送/替换文件。保留原有 OH 系统完整性。
  2. Remount /system — 部署前需要将 /system 分区重新挂载为读写模式 (mount -o rw,remount /)。
  3. 可回滚 — 替换系统文件前先备份为 .bak 文件，提供一键回滚脚本恢复原始状态。
  4. 分模块部署 — 支持 OH 系统服务、Android 运行时、适配层 三个模块独立部署和更新。
  5. 部署验证 — 每次部署后执行检查脚本，确认所有文件到位、权限正确、服务可启动。
注意：替换 OH 系统服务 .so 文件（如 libabilityms.z.so）具有一定风险，可能导致 OH 系统服务异常。建议先在 Phase 1 验证补丁不影响 OH 正常功能，确认无误后再进行后续部署。

2. 编译产物清单与部署位置

2.1 AOSP Java/DEX 产物

AOSP 框架层 Java 库，编译为 DEX 格式，放置在设备的 Android 框架目录下，供 ART VM 在启动时加载。

产物文件大小状态编译来源设备路径说明
framework.jar 39 MB ✅ AOSP 14 make framework /system/android/framework/ Android 核心框架库（含适配层补丁的 ActivityManager 等）
framework-classes.dex.jar 38 MB ✅ AOSP 14 make framework /system/android/framework/ Framework DEX 字节码（从 framework.jar 中提取的 classes.dex）
framework-res-package.jar 33 MB ✅ AOSP 14 make framework-res /system/android/framework/ Android 框架资源包（layouts, drawables, strings 等）
oh-adapter-framework.jar ~2 MB ❌ Adapter Android.bp /system/android/framework/ 适配层 Java 代码（Adapter 类、Bridge 类、Converter 类）
BOOTCLASSPATH 配置：这三个 JAR 文件都需要加入到 appspawn-x 的 BOOTCLASSPATH 环境变量中，ART VM 启动时会从此路径加载框架类。加载顺序为：framework.jar → framework-res-package.jar → oh-adapter-framework.jar。

2.2 OH 系统服务产物

应用了适配层补丁的 OH 系统服务动态库，需替换设备上的原始 .so 文件。这些补丁使 OH 系统服务能识别和管理通过适配层注册的 Android App 实例。

产物文件大小状态编译目标设备路径（替换）补丁内容
libabilityms.z.so 6.2 MB ✅ OH dayu210 /system/lib64/platformsdk/ AbilityManagerService：Android App 作为 Ability 的生命周期管理、Mission 分组
libappms.z.so 2.4 MB ✅ OH dayu210 /system/lib64/platformsdk/ AppMgrService：Android 进程识别、RemoteClientManager 扩展
libwms.z.so 1.5 MB ✅ OH dayu210 /system/lib64/platformsdk/ WindowManagerService：Android Window 属性映射
libbms.z.so 6.8 MB ✅ OH dayu210 /system/lib64/platformsdk/ BundleManagerService：APK 安装注册、BundleInfo 扩展
风险提示：这 4 个 .so 文件是 OH 核心系统服务库，替换后若有兼容性问题会导致对应系统服务无法启动。部署前必须备份原始文件，并在 Phase 1 中先验证补丁对 OH 原生功能无影响。

2.3 交叉编译产物（AOSP Native → musl）

Android 运行时（ART）及其依赖库，需要从 AOSP 源码交叉编译为 aarch64-linux-musl 目标（OH 使用 musl libc 而非 glibc/bionic）。这些库提供了 ART 虚拟机、JNI 环境和 Android 原生基础设施。

libdexfile.so 400 KB ❌ art/libdexfile/ /system/android/lib64/ DEX 文件解析库 — Layer 3 待完成
libartbase.so 312 KB ❌ art/libartbase/ /system/android/lib64/ ART 基础工具库（日志、文件操作、字符串）— Layer 3 待完成
libartpalette.so 22 KB ✅ art/libartpalette/ /system/android/lib64/ ART 平台抽象层（线程、信号处理适配）
libbionic_compat.so 17 KB ✅ 自建适配层 /system/android/lib64/ Bionic → musl 兼容层（property_get 等）
liblog.so 70 KB ✅ system/logging/ /system/android/lib64/ Android 日志库（重定向到 OH hilog）
libbase.so 140 KB ✅ system/libbase/ /system/android/lib64/ Android 基础库（strings、files、logging）
libcutils.so 39 KB ✅ system/core/libcutils/ /system/android/lib64/ Android C 工具库（properties、ashmem、trace）
libutils.so 169 KB ✅ system/core/libutils/ /system/android/lib64/ Android C++ 工具库（RefBase、String8/16、Looper）

交叉编译要点：OH 系统使用 musl libc，而 AOSP 原生库基于 bionic。交叉编译需要使用 OH SDK 自带的 aarch64-linux-ohos 工具链（基于 clang/LLVM），并通过 libbionic_compat.so 提供 bionic 特有 API 的兼容实现。

3. 部署脚本设计

3.1 deploy.sh 脚本结构

部署脚本 deploy.sh 位于项目根目录 D:\code\adapter\deploy.sh，支持按模块独立部署。

脚本使用方式

# 部署所有模块
./deploy.sh --target=all

# 仅部署 OH 系统服务 .so（替换原始文件）
./deploy.sh --target=oh-services

# 仅部署 Android 运行时（framework JAR + ART .so）
./deploy.sh --target=android-rt

# 仅部署适配层产物（appspawn-x + bridge .so + 配置文件）
./deploy.sh --target=adapter

# 检查部署是否完整
./deploy.sh --target=check

# 回滚所有替换的文件
./deploy.sh --target=rollback

脚本核心参数

参数说明
--target=<module>部署目标：oh-services | android-rt | adapter | all | check | rollback
--device=<id>指定设备 ID（默认使用 hdc list targets 发现的第一个设备）
--build-dir=<path>编译产物根目录（默认 ./out/）
--no-backup跳过备份（仅建议增量更新时使用）
--dry-run仅打印将要执行的命令，不实际执行
--reboot部署完成后自动重启设备

脚本内部结构

deploy.sh
├── 公共函数
│   ├── log_info() / log_warn() / log_error()   # 日志输出（带颜色）
│   ├── hdc_send()                               # 封装 hdc file send + 权限设置

│   ├── verify_file()                            # 验证文件大小和权限
│   └── check_device()                           # 检查设备连接状态
│
├── 部署模块
│   ├── deploy_oh_services()                     # OH 系统服务 .so 替换
│   ├── deploy_android_rt()                      # Android 运行时部署
│   ├── deploy_adapter()                         # 适配层产物部署

└── 入口

3.2 分模块部署流程

Target: oh-services（OH 系统服务替换）

替换设备上的 4 个 OH 系统服务动态库。这是风险最高的部署步骤，必须备份原始文件。

# 1. Remount 文件系统
hdc shell mount -o rw,remount /

hdc shell cp /system/lib64/platformsdk/libabilityms.z.so /system/lib64/platformsdk/libabilityms.z.so.bak
hdc shell cp /system/lib64/platformsdk/libappms.z.so /system/lib64/platformsdk/libappms.z.so.bak
hdc shell cp /system/lib64/platformsdk/libwms.z.so /system/lib64/platformsdk/libwms.z.so.bak
hdc shell cp /system/lib64/platformsdk/libbms.z.so /system/lib64/platformsdk/libbms.z.so.bak

# 3. 推送新的 .so 文件
hdc file send out/oh-services/libabilityms.z.so /system/lib64/platformsdk/libabilityms.z.so
hdc file send out/oh-services/libappms.z.so /system/lib64/platformsdk/libappms.z.so
hdc file send out/oh-services/libwms.z.so /system/lib64/platformsdk/libwms.z.so
hdc file send out/oh-services/libbms.z.so /system/lib64/platformsdk/libbms.z.so

# 4. 设置权限
hdc shell chmod 644 /system/lib64/platformsdk/libabilityms.z.so
hdc shell chmod 644 /system/lib64/platformsdk/libappms.z.so
hdc shell chmod 644 /system/lib64/platformsdk/libwms.z.so
hdc shell chmod 644 /system/lib64/platformsdk/libbms.z.so

# 5. 重启设备使 OH 服务加载新的 .so
hdc shell reboot

hdc file send out/android-rt/libartpalette.so /system/android/lib64/libartpalette.so
hdc file send out/android-rt/libbionic_compat.so /system/android/lib64/libbionic_compat.so
hdc file send out/android-rt/liblog.so /system/android/lib64/liblog.so
hdc file send out/android-rt/libbase.so /system/android/lib64/libbase.so
hdc file send out/android-rt/libcutils.so /system/android/lib64/libcutils.so

hdc shell chmod 644 /system/android/framework/*.jar

Target: adapter（适配层产物部署）

部署适配层自身的二进制文件和配置文件。

# 1. 推送 appspawn-x 可执行文件
hdc file send out/adapter/appspawn-x /system/bin/appspawn-x
hdc shell chmod 755 /system/bin/appspawn-x

# 2. 推送适配层 JNI 库
hdc file send out/adapter/liboh_adapter_bridge.z.so /system/lib64/liboh_adapter_bridge.z.so

hdc shell chmod 644 /system/lib64/liboh_adapter_bridge.z.so
hdc shell chmod 644 /system/lib64/libapk_installer.z.so

# 3. 推送配置文件
hdc file send out/adapter/appspawn_x.cfg /system/etc/init/appspawn_x.cfg
hdc file send out/adapter/appspawn_x_sandbox.json /system/etc/appspawn_x_sandbox.json
hdc shell chmod 644 /system/etc/init/appspawn_x.cfg
hdc shell chmod 644 /system/etc/appspawn_x_sandbox.json

4.2 appspawn-x 4 阶段初始化

appspawn-x 启动后执行 4 个阶段的初始化，最终进入事件循环等待 Foundation 的 spawn 请求。

appspawn-x main() | ┌───────────────┼───────────────┐ | | | Phase 1: OH 安全 Phase 2: ART VM Phase 3: 预加载 | | | v v v ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ | sandbox 配置 | | JniInvocation | | preload() | | SELinux 初始化 | | dlopen libart | | ~7000 classes| | AccessToken | | CreateJavaVM | | resources | | listen socket | | registerJNI | | shared libs | └──────────────┘ └──────────────┘ | adapter bridge| | GC | └──────────────┘ | v Phase 4: 事件循环 ┌──────────────┐ | poll() 等待 | | Unix socket | | signal ready | └──────────────┘ bootevent.appspawn_x.ready = true

Phase 1: OH 安全初始化

  • 读取沙箱配置 — 解析 /system/etc/appspawn_x_sandbox.json，准备好 mount 规则模板
  • SELinux 初始化 — 加载 SELinux 策略，确认 appspawn-x 进程运行在 u:r:appspawn_x:s0 域
  • AccessToken 初始化 — 连接 OH AccessTokenManager 服务，用于后续给子进程分配 TokenID
  • 创建 listen socket — 从 init 继承或创建 /dev/unix/socket/appspawn_x Unix domain socket

Phase 2: ART VM 初始化

  • JniInvocation::Init() — 通过 libnativehelper.so 初始化 JNI 调用框架，dlopen libart.so
  • JNI_CreateJavaVM() — 创建 ART 虚拟机实例，配置堆大小、GC 策略、BOOTCLASSPATH 等
  • RegisterNativeMethods() — 注册 ART 核心 JNI 方法（Thread、Runtime、System 等）
  • VM 参数示例：-Xms8m -Xmx512m -XX:HeapGrowthLimit=256m -Xbootclasspath:${BOOTCLASSPATH}

Phase 3: 预加载

  • AppSpawnXInit.preload() — 通过 JNI 调用 Java 侧的预加载入口
  • 类预加载 — 加载约 7000 个常用 Android 框架类到内存，fork 后子进程可通过 COW 共享
  • 资源预加载 — 加载 framework-res-package.jar 中的系统资源
  • 共享库加载 — 预加载 liboh_adapter_bridge.z.so，注册适配层 JNI 方法
  • GC 清理 — 预加载完成后执行 Full GC，减少 fork 后的 COW 页面数

Phase 4: 事件循环

  • poll() 等待 — 在 Unix socket 上阻塞等待 Foundation 发来的 SpawnMsg
  • 信号 bootevent — 设置 bootevent.appspawn_x.ready = true，通知 OH init 服务就绪
  • Foundation 的 AbilityManagerService 会等待此 bootevent 后才发送 spawn 请求

4.3 App 启动流程

从 OH Foundation 发起 StartAbility 到 Android App 的 ActivityThread.main() 开始运行的完整流程。

OH Foundation (AbilityManagerService) | 1. 构造 SpawnMsg (bundleName, uid, gid, flags, abilityName...) | 2. send(SpawnMsg) via Unix socket | appspawn-x (事件循环收到请求) | 3. fork() ─────────────────────────────────────┐ | | 父进程: 回到 poll() 子进程: | 4. 子进程特化 (child_main) ├─ DAC: setuid(uid), setgid(gid) ├─ Sandbox: unshare(CLONE_NEWNS) │ └─ mount 沙箱规则 ├─ SELinux: setcon(app_domain) └─ AccessToken: SetSelfTokenID(tokenId) | 5. OHEnvironment.initialize() ├─ loadLibrary("oh_adapter_bridge") ├─ nativeInit() → 连接 OH 系统服务 │ ├─ OH AbilityManager IPC proxy │ ├─ OH AppMgr IPC proxy │ ├─ OH WindowManager IPC proxy │ └─ OH BundleManager IPC proxy └─ isOHEnvironment() → true | 6. AppSpawnXInit.initChild(bundleName, abilityName) ├─ 设置 classloader ├─ 设置 process name └─ 调用 ActivityThread.main() | 7. ActivityThread.main() ├─ Looper.prepareMainLooper() ├─ new ActivityThread() ├─ attach(false) → 通知 AMS 进程启动 │ └─ ActivityManagerAdapter → OH AppMgr └─ Looper.loop() → 进入消息循环

关键步骤详解

Step 3: fork()

appspawn-x 调用 fork() 创建子进程。由于 Phase 3 已完成预加载，子进程通过 COW（Copy-On-Write）共享父进程已加载的约 7000 个 Java 类和 ART VM 内部数据结构，避免重复初始化的开销。

Step 4: 子进程特化

  • DAC（自主访问控制）：setuid() / setgid() 降权到目标 App 的 UID/GID
  • Sandbox（沙箱隔离）：unshare(CLONE_NEWNS) 创建独立 mount namespace，按 appspawn_x_sandbox.json 规则挂载
  • SELinux：setcon() 切换到 App 专用的 SELinux 安全域
  • AccessToken：SetSelfTokenID() 设置 OH AccessToken，App 后续的 IPC 调用权限由此 Token 控制

Step 5: OHEnvironment.initialize()

在 Java 侧初始化 OH 适配环境，加载 JNI 库并通过 nativeInit() 连接所有 OH 系统服务的 IPC 代理。此后所有 Android 系统 API 调用将通过适配层路由到 OH 服务。

4.4 App 运行时 IPC 流程

Android App 启动后，其系统调用通过以下三条主要 IPC 路径与 OH 系统服务交互。

正向桥接：Android App → OH 系统服务

Android App (Java) | ActivityManager.getService() | ActivityManagerAdapter (extends IActivityManager.Stub) | Java → native JNI: nativeStartAbility(want, options) | oh_ability_manager_client.cpp | OH IPC (MessageParcel) OH AbilityManagerService | [Start Ability / Service / Content Provider ...]

反向回调：OH 系统服务 → Android App

OH AbilityManagerService | OH IPC callback IAppScheduler proxy (OH → appspawn-x 子进程) | app_scheduler_adapter.cpp | C++ → JNI callback AppSchedulerBridge.java | IApplicationThread (Android) | ActivityThread.handleXxx() | [scheduleTransaction / scheduleTrimMemory / ...]

Surface 桥接：图形渲染

Android App (Canvas / View) | Android Surface → IGraphicBufferProducer | oh_graphic_buffer_producer.cpp | buffer 映射 OH Surface (producer) | oh_surface_bridge.cpp | RS API OH RSSurfaceNode → RSUIDirector | OH IPC OH RenderService (GPU 合成 + 显示)

5.4 沙箱配置详解

appspawn_x_sandbox.json 定义了 Android App 子进程的 mount namespace 规则，确保 App 只能访问授权的文件系统路径。

{
    "common": {
        "mount-paths": [
            {
                "src-path": "/system/android/framework",
                "sandbox-path": "/system/framework",
                "sandbox-flags": ["bind", "rec", "ro"],
                "comment": "Android framework JAR 映射到标准路径"
            },
            {
                "src-path": "/system/android/lib64",
                "sandbox-path": "/system/lib64/android",
                "sandbox-flags": ["bind", "rec", "ro"],
                "comment": "ART native 库映射"
            },
            {
                "src-path": "/data/misc/appspawnx/dalvik-cache",
                "sandbox-path": "/data/dalvik-cache",
                "sandbox-flags": ["bind", "rec", "rw"],
                "comment": "DEX 优化缓存"
            }
        ],
        "mount-groups": [
            {
                "name": "app-data",
                "src-path": "/data/app/${bundleName}",
                "sandbox-path": "/data/data/${packageName}",
                "sandbox-flags": ["bind", "rec", "rw", "nosuid"],
                "comment": "App 私有数据目录"
            },
            {
                "name": "app-cache",
                "src-path": "/data/app/${bundleName}/cache",
                "sandbox-path": "/data/data/${packageName}/cache",
                "sandbox-flags": ["bind", "rec", "rw", "nosuid", "nodev"],
                "comment": "App 缓存目录"
            }
        ]
    },
    "individual": {
        "comment": "特定 App 的额外 mount 规则（如需访问外部存储等）"
    }
}

沙箱规则说明

规则类别说明
common/mount-paths 所有 Android App 共享的 mount 规则，将 OH 设备上的实际路径映射到 Android 标准路径
common/mount-groups 按 App 实例化的 mount 规则，${bundleName} 和 ${packageName} 在 fork 时替换为具体值
individual 特定 App 的定制化规则（如需要访问 sdcard 等外部存储的 App）
sandbox-flags bind（绑定挂载）、rec（递归）、ro（只读）、rw（读写）、nosuid（禁止 SUID）、nodev（禁止设备文件）

B.4.4 华为云服务器省钱技巧

  • 不用时关机：按量付费关机后 CPU/内存不计费，仅磁盘计费
  • 不要释放实例：关机 ≠ 释放。释放会删除所有数据
  • 编译完立即关机：增量编译通常 5-10 分钟完成
  • 善用 ccache：首次全量编译后，后续增量编译时间大幅缩短
  • 按需升降配：日常可用 8核16G，全量编译时临时升级到 16核32G
