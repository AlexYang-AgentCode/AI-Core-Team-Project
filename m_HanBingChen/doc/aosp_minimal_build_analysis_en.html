1. Background and Objectives

1.1 Background

The core architecture of the Android-OH adapter project is: running Android Apps on an OH system while retaining the Android Framework client-side code and ART virtual machine, but replacing system services with OH implementations. Therefore, we need to build a subset of AOSP containing only the App runtime, excluding system_server and all the system services it hosts.

This report analyzes the source code structure and build system of AOSP 14 (android-14.0.0_r1), identifies which modules need to be compiled and which can be excluded, and provides build strategy recommendations.

1.2 Trimming Objectives

To Retain (Runs Within App Process)

  • ART Virtual Machine — Dex bytecode loading and execution, JIT/AOT compilation, GC, JNI
  • Framework Client Code — ActivityThread, View system, Intent, all AIDL-generated Stub/Proxy classes
  • Framework Native Libraries — libandroid_runtime (JNI bridge), Binder client, Surface client
  • Java Core Libraries — java.lang.*, java.util.*, java.io.*, etc.
  • C Runtime — libc, libm, dynamic linker
  • System Resources — framework-res.apk

To Exclude (Replaced by OH System Services)

  • system_server process and all system services (AMS, WMS, PMS, etc.)
  • System apps (SystemUI, Launcher, Settings, etc.)
  • HAL service layer

2. AOSP 14 Build System Analysis

2.1 AOSP Source Top-Level Structure

The AOSP 14 source code on the server is located at /root/aosp/, containing the following core directories:

/root/aosp/ ├── art/ # ART virtual machine runtime ├── bionic/ # Android C runtime (libc, libm, linker) ├── build/ # Build system (Soong/Blueprint) ├── external/ # Third-party dependency libraries ├── frameworks/ # Android Framework │ ├── base/ # Core framework (Java + JNI + resources) │ └── native/ # Native framework libraries (Binder, Surface, Input) ├── libcore/ # Java core libraries (java.lang.*, java.util.*) ├── libnativehelper/ # JNI helper library ├── prebuilts/ # Prebuilt toolchains (Clang, JDK, SDK tools) └── system/ # System base libraries (libbase, liblog, libutils)

2.2 Framework Build Target Hierarchy

The AOSP 14 Framework build defines a clear hierarchy in frameworks/base/Android.bp:

LevelBuild TargetTypeDescription
1 framework-core-sources filegroup Core source collection: all *.java + *.aidl files under core/java/, including android.app.*, android.view.*, android.content.*, android.os.*, and all other client APIs
2 framework-non-updatable-sources filegroup Aggregates framework-core-sources + domain sources (graphics, telephony, media, wifi, location, opengl, etc.) + native/HAL AIDL interfaces
3 framework-minus-apex-defaults java_defaults Build configuration: compiles framework-non-updatable-sources, enables AIDL processing (generate_get_transaction_name: true)
4 framework-minus-apex java_library
(installable)		 The actual framework.jar, installed on device. Uses stem: "framework". This is the client-side Framework — contains no system service code
5 framework java_library
(not installable)		 Compile-time dependency target: framework-minus-apex + framework-updatable-stubs. Used only for other modules to reference during compilation
Key Finding: framework-minus-apex is the final artifact framework.jar, and it contains only client-side code. System services (services.jar) are a completely independent build target, defined in frameworks/base/services/Android.bp. There is no compile-time dependency between the two.

2.3 Natural Client-Server Separation

The AOSP 14 build system naturally separates the App runtime and system services into two independent JARs:

┌─────────────────────────────────────────────────────────────────────────┐ │ AOSP 14 Build Targets │ │ │ │ ┌─ Client Side (Runs Within App Process) ─────────────────────────┐ │ │ │ │ │ │ │ framework-minus-apex ──→ framework.jar │ │ │ │ │ │ │ │ Contains: │ │ │ │ • android.app.* (ActivityThread, Application, Intent, ...) │ │ │ │ • android.view.* (View, ViewGroup, Window, Surface, ...) │ │ │ │ • android.content.* (ContentProvider, ContentResolver, ...) │ │ │ │ • android.os.* (Binder, IBinder, Parcel, Handler, Looper, ...) │ │ │ │ • All AIDL-generated Stub/Proxy classes │ │ │ │ • com.android.internal.* (internal utility classes) │ │ │ │ │ │ │ │ Does NOT contain: any *Service implementation classes │ │ │ │ │ │ │ └──────────────────────────────────────────────────────────────────┘ │ │ │ │ ┌─ Server Side (Runs in system_server) ── NOT needed ─────────────┐ │ │ │ │ │ │ │ services ──→ services.jar │ │ │ │ │ │ │ │ Contains: │ │ │ │ • ActivityManagerService, WindowManagerService │ │ │ │ • PackageManagerService, PowerManagerService │ │ │ │ • SystemServer.java (entry point) │ │ │ │ • services.core, services.accessibility, services.autofill, ... │ │ │ │ │ │ │ └──────────────────────────────────────────────────────────────────┘ │ └─────────────────────────────────────────────────────────────────────────┘
Conclusion: The AOSP build system has already done the separation for us. Building framework-minus-apex yields a framework.jar containing only client-side code, requiring no additional trimming. The services target does not need to be built at all.

JNI Layer (core/jni/)

libandroid_runtime (cc_library_shared) is the Framework's Java↔Native bridge library, containing:

  • android_app_ActivityThread.cpp — ActivityThread native methods
  • android_view_Surface*.cpp — Surface-related JNI
  • android_view_InputEventReceiver.cpp — Input event receiving
  • android_graphics_*.cpp — Graphics system JNI
  • AndroidRuntime.cpp — ART VM initialization and JNI registration entry point

Resources (core/res/)

Produces framework-res.apk, containing system default themes, layouts, strings, etc. Required during appspawn-x preloading.

5.2 System Apps (packages/)

All preinstalled system apps are replaced by their OH counterparts:

  • packages/apps/Launcher3/ → OH Launcher
  • frameworks/base/packages/SystemUI/ → OH SystemUI
  • packages/apps/Settings/ → OH Settings
  • packages/apps/Contacts/ → OH Contacts
  • packages/apps/Dialer/ → OH Dialer
  • All other packages/apps/*

5.3 HAL Layer (hardware/)

The Android HAL (Hardware Abstraction Layer) is replaced by OH's HDI (Hardware Device Interface). This includes:

  • Graphics HAL (gralloc, HWC) → OH Graphics HDI
  • Audio HAL → OH Audio HDI
  • Camera HAL → OH Camera HDI
  • Sensors HAL → OH Sensors HDI

Nothing under the hardware/ directory needs to be compiled.

6.3 Relationship Between app_process and appspawn-x

In standard AOSP, App processes are forked from Zygote, with the entry point being app_process64 (frameworks/base/cmds/app_process/app_main.cpp). In the adapter project:

  • app_process64 is replaced by appspawn-x
  • appspawn-x's ART initialization logic is based on AndroidRuntime::start() from app_main.cpp
  • app_process64 can serve as an optional build target for debugging and comparison verification
  • appspawn-x itself is built via the OH build system (BUILD.gn), not through the AOSP build

7. Build Strategy

7.1 Option A: Module-Level On-Demand Build

Use the standard AOSP build system, compiling only the required modules:

# Initialize the build environment
source build/envsetup.sh
lunch aosp_arm64-eng

# Build core modules (dependencies resolved automatically)
m framework-minus-apex     # → framework.jar (client-side Framework)
m libandroid_runtime       # → libandroid_runtime.so (JNI bridge)
m framework-res            # → framework-res.apk (system resources)
m dex2oat                  # → dex2oat (AOT compiler)

# Adapter layer modules
m liboh_adapter_bridge     # → liboh_adapter_bridge.so
m oh-adapter-framework     # → oh-adapter-framework.jar

# ART / bionic / libcore are compiled automatically as dependencies, no explicit specification needed

Advantages

  • Simple and straightforward, leverages AOSP's existing dependency resolution
  • No need to create custom makefiles
  • Quick to verify

Disadvantages

  • First build still takes considerable time (long dependency chain)
  • May pull in unnecessary dependency modules
  • lunch aosp_arm64-eng configures a full device product, some configurations may not apply

7.2 Option B: Custom Product

Create a minimal Product makefile to precisely control the build scope:

# device/adapter/oh_adapter/AndroidProducts.mk
PRODUCT_MAKEFILES := \
    $(LOCAL_DIR)/oh_adapter.mk

COMMON_LUNCH_CHOICES := \
    oh_adapter-eng

# device/adapter/oh_adapter/oh_adapter.mk
$(call inherit-product, build/target/product/core_64_bit_only.mk)

PRODUCT_NAME := oh_adapter
PRODUCT_DEVICE := generic_arm64
PRODUCT_BRAND := Android

# Include only artifacts required for App runtime
PRODUCT_PACKAGES += \
    framework \
    framework-res \
    libandroid_runtime \
    core-oj \
    core-libart \
    libart \
    dex2oat \
    oh-adapter-framework \
    liboh_adapter_bridge

# Exclude system services and system apps
PRODUCT_PACKAGES_REMOVE += \
    SystemUI \
    Launcher3 \
    Settings

Advantages

  • Precise control over build scope, excludes unnecessary modules
  • Faster build times (reduces unnecessary builds)
  • Better suited for long-term maintenance
  • Can define custom lunch targets

Disadvantages

  • Requires understanding of the AOSP Product build system
  • Higher initial configuration effort
  • Need to handle implicit product configurations in the dependency chain

7.3 Comparison and Recommendation

DimensionOption A (On-Demand Build)Option B (Custom Product)
Startup CostLow (ready to use)Medium (requires creating mk files)
Build SpeedMedium (may pull extra dependencies)Fast (precise scope control)
MaintainabilityLow (no centralized configuration)High (centralized product definition)
ReproducibilityMedium (depends on manually specifying targets)High (just lunch + make)
Applicable PhaseEarly validationOngoing development
Recommendation: Start with Option A to quickly validate the build process and artifact completeness, then migrate to Option B to establish a formal build configuration once verified.

9. Conclusion

Key Conclusions

  1. The AOSP build system natively supports client/server separation — framework.jar (client) and services.jar (server) are completely independent build targets. The client portion can be built without any additional trimming.
  2. Build scope is well-defined — 9 core modules need to be built (Framework client, ART, libcore, bionic, native framework libraries, system base libraries, JNI helper library, adapter layer, prebuilt toolchains), while system services, system apps, and the HAL layer are excluded.
  3. AOSP modification points are concentrated — All 4 source file modifications are within frameworks/base/core/java/, which falls within the scope of modules that must be compiled.
  4. Server environment is ready — The oh-build server (16 cores, 60GB memory, 378GB available disk) fully meets build requirements.
  5. Recommended two-phase approach — Start with Option A (module-level on-demand build) for quick validation, then migrate to Option B (custom Product) to establish a formal build process.
