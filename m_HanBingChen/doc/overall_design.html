1. 项目概述

1.1 项目背景与目标

本项目的目标是在 OpenHarmony（OH）系统上运行未经修改的 Android 应用程序。核心思路是在 Android Framework 与 OH 系统服务之间构建一个适配层（Adapter Layer），拦截 Android App 通过 Binder IPC 发出的系统服务调用，将其转换并路由到等价的 OH 系统服务接口。

这种方式不同于完整的 Android 容器或虚拟机方案：我们保留 Android 运行时（ART）和 Framework 层，但将底层系统服务替换为 OH 实现。Android App 无感知地运行在 OH 内核和系统服务之上。

目标硬件：HH-SCDAYU600（DAYU600）开发板，搭载紫光展锐 UIS7885（ums9620）SoC。

1.2 核心设计理念

理念一：App 与系统服务接口桥接

适配层的核心职责是实现 Android App 与 OH 系统服务之间的双向透明通信。正向方向上，拦截 Android App 通过 Binder IPC 发出的系统服务调用（如 ActivityManager、WindowManager、PackageManager），通过适配器将其转换为对等价 OH 系统服务（如 AbilityManager、SceneSessionManager、BundleManager）的调用，App 无需任何修改即可透明使用 OH 系统能力。反向方向上，OH 系统服务主动推送的回调（如 IAbilityScheduler 生命周期通知、ISessionStage 窗口事件、IAbilityConnection 服务连接通知）经过回调桥接层，转换为 Android App 能够识别的标准回调接口（如 IApplicationThread、IWindow、IServiceConnection），确保 App 的生命周期管理、窗口交互和服务绑定机制正常运作。整个桥接过程对 App 完全透明，530 个正向方法和 210 个反向回调方法均被覆盖。

理念二：Framework 自渲染

保留完整的 Android Framework 渲染管线，App 的 UI 渲染完全由 Android Framework 自身完成，适配层不介入渲染过程本身。Android 的 View 体系（Measure → Layout → Draw）、RenderThread 硬件加速渲染、Skia 2D 图形库、HWUI 硬件抽象层等组件原封不动地在 ART 中运行。渲染产生的 GraphicBuffer 通过 Surface 桥接模块（OHGraphicBufferProducer）提交给 OH RenderService 进行合成显示。Android 和 OH 的图形缓冲区均基于 Linux dmabuf 机制，BufferQueue 状态机可 1:1 映射（FREE↔RELEASED、DEQUEUED↔REQUESTED、QUEUED↔FLUSHED、ACQUIRED↔ACQUIRED），保证了跨运行时缓冲区共享的可行性。这一设计避免了重新实现渲染管线的巨大工程量，同时保证了 Android App 的原生渲染质量和性能。

理念三：类继承桥接

所有正向适配器均采用类继承方式（extends IXxx.Stub）实现，而非动态代理（Dynamic Proxy）。每个适配器直接继承 AIDL 自动生成的 Stub 抽象类，显式覆写每一个接口方法，获得编译时的完整类型检查和方法签名验证。相比动态代理方案（通过 InvocationHandler 拦截所有调用，以方法名字符串匹配分发），类继承方式消除了运行时反射开销，支持 IDE 代码导航和安全重构，且在接口变更时（如 AOSP 版本升级）编译器会立即报错，避免运行时才暴露的兼容性问题。每个适配器声明自己的 native 方法用于 OH 服务访问，持有独立的 OH 服务句柄（如 long mOhAbilityManager），保持模块间的清晰边界。

理念四：App 孵化进程 — 向下顺从 OH 机制，向上包含 ART 虚拟机

Android App 的进程孵化由 appspawn-x 混合孵化器负责，其设计遵循"向下顺从、向上包含"的原则。向下，appspawn-x 完全遵循 OH 的进程管理机制：由 OH foundation 统一调度，使用 OH 标准的 socket 协议接收 spawn 请求，fork 后的子进程严格执行 OH 安全特化流程（UID/GID 设置、SELinux 上下文切换、AccessToken 绑定、sandbox 挂载），确保 Android App 进程与 OH 原生进程在安全模型和进程管控上完全一致，不引入任何安全模型的特例或旁路。向上，appspawn-x 在进程中集成了完整的 ART（Android Runtime）虚拟机，支持 Dex 字节码的加载与执行。ART 提供 JIT/AOT 编译、垃圾回收、JNI 调用等核心能力；APK 中的 classes.dex 通过 dex2oat 预编译为本地机器码（.odex/.vdex），确保运行时性能。appspawn-x 在 fork 前预加载 ART VM 及约 7000 个 Framework 核心类和资源，利用 COW（Copy-On-Write）机制实现多进程间内存共享，显著降低 App 启动延迟和内存占用。子进程 fork 后进入 ActivityThread.main() → Looper.loop()，在 ART 中运行标准的 Android 应用主循环。

理念五：Java-C++ 双语言适配

适配层采用 Java 与 C++ 双语言协作实现，通过 JNI（Java Native Interface）连接。这一选择由两端系统的技术栈决定：Android 的 AIDL 接口、Framework 数据类型（Intent、Bundle、Parcelable 等）和组件机制（ClientTransaction、ActivityRecord 等）均为 Java 实现，适配层的 Stub 继承和 Android 侧交互必须使用 Java；而 OH 系统服务的 IPC 接口基于 C++ 的 IRemoteProxy/IRemoteBroker 机制，适配层与 OH 服务的通信必须使用 C++。两种语言通过 liboh_adapter_bridge.so 共享库连接——正向调用中，Java 适配器通过 native 方法调用 C++ 层发起 OH IPC；反向回调中，C++ Stub 通过 AttachCurrentThread 获取 JNIEnv 并调用 Java Bridge 方法，每个 C++ 适配器缓存 Java 对象引用（jobject）和方法 ID（jmethodID），避免重复查找开销。

理念六：最小 OH 修改原则

优先在适配层内部解决所有兼容性问题，仅在确实无法绕过时才修改 OH 系统源码。所有 OH 侧修改以补丁文件形式管理（存放于 ohos_patches/ 目录），方便追踪、审查和随 OH 版本升级同步维护。当前 OH 修改范围控制在两个子系统内：ability_rt（能力管理框架，5 个文件）用于支持 Mission 内 Ability 栈和 Android 进程管理；bundle_framework（包管理框架，4 个文件）用于支持 APK 安装和 Android 包类型识别。其余所有适配逻辑（接口转换、数据格式映射、回调桥接等）均在适配层内闭合完成，OH Launcher、SystemUI 等上层应用无需任何修改。

2.2 分层架构

整个适配方案由以下五个层次组成：

第 1 层：Android 应用层

标准 Android App（Activity、Service、BroadcastReceiver、ContentProvider），使用标准 Android SDK API，无需任何修改即可运行。

第 2 层：Android Framework 层（修改的 AOSP 14）

在关键服务获取点（ActivityManager、ActivityTaskManager、WindowManagerGlobal、ActivityThread）插入环境检测逻辑：OHEnvironment.isOHEnvironment() 为 true 时，返回适配器实例替代原始 Binder 代理。共修改 4 个 AOSP 源文件。

第 3 层：适配层（本项目核心）

位于 framework/ 目录。包含正向适配器（5 个 Stub 子类）、反向回调桥接（6 个 Bridge 类 + 5 个 C++ Stub 适配器）、数据转换器（Intent↔Want、Broadcast↔CommonEvent、URI 映射等）、以及 JNI 层（liboh_adapter_bridge.so）。

第 4 层：OH 系统服务层

标准 OH 系统服务（AbilityManagerService、AppMgrService、WindowManagerService、RenderService、CommonEventService、DataShareService、BundleManagerService）。仅在必要时通过补丁扩展（ohos_patches/）。

第 5 层：进程管理层（appspawn-x）

混合孵化器，结合 OH 的安全模型和 Android Zygote 的 ART 预加载。负责 Android App 进程的创建和初始化。

3.2 接口可行性分析结论

对 Android 4 大系统服务接口（530 正向方法）和 6 大回调接口（181 反向方法）进行了逐方法可行性分析：

4. 正向桥接设计

4.1 桥接模式：类继承

所有正向适配器均采用类继承模式（extends IXxx.Stub），而非动态代理。这一选择基于以下考虑：

对比项类继承（选用）动态代理（弃用）
类型安全编译时检查所有方法签名运行时才发现参数类型错误
性能直接方法调用，无反射开销每次调用经过 InvocationHandler 反射
可维护性IDE 可导航，重构安全方法名字符串匹配，易出错
代码量较多（需显式覆写每个方法）较少（统一转发逻辑）

每个适配器的基本结构：

public class ActivityManagerAdapter extends IActivityManager.Stub {
    private long mOhAbilityManager;   // native OH 服务句柄
    private long mOhAppMgr;           // 可持有多个 OH 服务句柄

    public ActivityManagerAdapter() {
        mOhAbilityManager = nativeConnectAbilityManager();
        mOhAppMgr = nativeConnectAppMgr();
    }

    // [BRIDGED] — 已实现到 OH 的完整映射
    @Override
    public int startActivity(...) {
        Want want = IntentWantConverter.intentToWant(intent);
        return nativeStartAbility(mOhAbilityManager, want);
    }

    // [STUB] — 记录调用，返回默认值
    @Override
    public void setVrMode(...) { logStub("setVrMode"); }

    private native long nativeConnectAbilityManager();
    private native int nativeStartAbility(long handle, Want want);
}

5. 反向回调桥接设计

5.1 回调方向与线程模型

反向回调是 OH 系统服务主动推送到 App 的调用（如生命周期转换、窗口配置变化）。由于 OH 使用 C++ Binder，回调到达时在 OH Binder 线程上，需要经过以下路径才能到达 Android App：

5.3 两层架构：C++ Stub + Java Bridge

反向回调采用两层架构的原因：

  • C++ Stub 层：继承 OH 的 IRemoteStub 子类，处理 OH Binder OnRemoteRequest 消息解析。这一层必须用 C++ 实现，因为 OH Binder 是 C++ 原生接口。
  • Java Bridge 层：持有 Android IApplicationThread / IWindow / IServiceConnection 引用，负责将 OH 回调数据转换为 Android API 调用。这一层用 Java 实现，因为 Android Binder 回调接口是 Java AIDL 生成的。
  • JNI 连接：C++ Stub 通过 AttachCurrentThread 获取 JNIEnv，调用 Java Bridge 对应方法。每个 C++ 适配器缓存 Java Bridge 的 jobject 和关键方法的 jmethodID。

7.1 Activity ↔ Ability 适配

12. 构建系统

项目采用 AOSP 构建系统（Android.bp）和 OH 构建系统（BUILD.gn）双线构建，按产物类型严格分工：

构建分工原则：AOSP 构建仅负责平台无关产物（Java 字节码、资源、APK），所有 Native C++ 产物统一由 OH 构建系统编译。

原因分析：
  • 运行环境约束 — 所有 native 库最终运行在 OH 系统上（musl libc），必须使用 OH 工具链（clang + musl）编译。AOSP Soong 生成的 native 产物链接 bionic libc，无法在 OH 系统上运行。
  • OH IPC 依赖 — liboh_adapter_bridge.so 需要链接 OH IPC 库（ipc_core、samgr_proxy、ability_manager 等），这些库仅在 OH 构建环境中可用。此前 Android.bp 中 OH 依赖被注释掉，无法真正链接。
  • Java 产物平台无关 — .jar 文件是 Dex 字节码，由 ART VM 解释执行，不依赖底层 libc 实现。.apk 包含字节码和资源，同样平台无关。因此 AOSP 构建（Soong + AIDL 编译器）仍是编译 Java 层的最佳选择。

AOSP 构建（Android.bp）— 平台无关产物
