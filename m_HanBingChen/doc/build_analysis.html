1. 分析背景与目标

1.1 项目编译架构总览

Android-OH 适配项目的核心架构是：在 OH 系统上运行 Android App，保留 Android Framework 客户端代码和 ART 虚拟机，但将系统服务替换为 OH 实现。适配层（Adapter）作为桥梁，拦截 Android App 发出的 Binder IPC 调用，转换为 OH 系统服务的 IPC 请求。

这一架构决定了项目需要两套独立的编译体系协同工作：

维度AOSP 侧OH 侧
编译内容 Framework 客户端代码（framework.jar）
ART 虚拟机 / C 运行时		 系统服务端代码（各 SA 的 .z.so）
IPC 客户端库（inner_kits）
排除内容 system_server 及所有系统服务 OH 的 App 框架层（ArkUI、NAPI 等）
构建系统 Soong (Android.bp / Blueprint) GN + Ninja (BUILD.gn)
产物用途 App 进程内运行 适配层 JNI 库链接 + OH 系统服务运行
源码修改 4 个 AOSP 文件（aosp_patches/） 2 个 OH 模块多个文件（ohos_patches/）
关键区别：AOSP 侧只需编译客户端（排除服务端），因此是"裁剪"。OH 侧需要完整的系统服务在设备上运行，同时适配层还需要链接 OH 的 IPC 客户端库（inner_kits），因此是"按需选择 + 补丁增强"。

1.2 构建隔离原则

关键规则：构建过程中不得直接修改 AOSP 或 OH 的源代码和构建配置文件。
内容存放位置说明
AOSP 源码修改adapter/aosp_patches/.patch 文件或整文件替换
OH 源码修改adapter/ohos_patches/.patch 文件或整文件替换
OH 构建系统补丁adapter/ohos_patches/build/GN/gni 文件的补丁
编译配置和脚本adapter/build/构建脚本、产品配置等
编译产出adapter/out/.z.so、可执行文件等

构建流程应为：应用补丁到工作副本 → 编译 → 收集产物到 adapter/out/ → 还原工作副本。AOSP 和 OH 源码仓库始终保持干净。

2. 构建体系分析

2.1 AOSP 14 构建体系（Soong / Android.bp）

AOSP 14 使用 Soong 构建系统，以 Android.bp（Blueprint 格式）定义模块。Soong 将所有 Android.bp 文件解析为一张完整的依赖图，然后生成 Ninja 构建文件执行编译。

AOSP 源码顶层结构

/root/aosp/ ├── art/ # ART 虚拟机运行时 ├── bionic/ # Android C 运行时 (libc, libm, linker) ├── build/ # 构建系统 (Soong/Blueprint) ├── external/ # 第三方依赖库 ├── frameworks/ # Android Framework │ ├── base/ # 核心框架 (Java + JNI + 资源) │ └── native/ # 原生框架库 (Binder, Surface, Input) ├── libcore/ # Java 核心库 (java.lang.*, java.util.*) ├── libnativehelper/ # JNI 辅助库 ├── prebuilts/ # 预编译工具链 (Clang, JDK, SDK tools) └── system/ # 系统基础库 (libbase, liblog, libutils)

Framework 构建目标层次

层次构建目标类型说明
1 framework-core-sources filegroup 核心源码集合：core/java/ 下所有 *.java + *.aidl 文件
2 framework-non-updatable-sources filegroup 聚合 framework-core-sources + 领域源码（graphics、telephony、media 等）
3 framework-minus-apex-defaults java_defaults 编译配置：编译 framework-non-updatable-sources，启用 AIDL 处理
4 framework-minus-apex java_library (installable) 实际的 framework.jar，使用 stem: "framework"。仅含客户端 Framework
关键发现：framework-minus-apex 即为最终产物 framework.jar，它仅包含客户端代码。系统服务（services.jar）是完全独立的构建目标。

2.2 OH 6.1 构建体系（GN / BUILD.gn）

OpenHarmony 使用 GN（Generate Ninja）作为构建元语言，生成 Ninja 构建文件后由 Ninja 执行实际编译。

OH 源码顶层结构

/root/oh/ # OpenHarmony 6.1-Release (~78 GB) ├── build/ # 构建系统 (GN + Ninja + Python 脚本) ├── foundation/ # 系统能力层（核心系统服务） │ ├── ability/ability_runtime/ # AMS, AppMgr, Mission 服务 ★ │ ├── window/window_manager/ # WMS, SceneSession, DisplayMgr ★ │ ├── graphic/graphic_2d/ # RenderService, RSUIDirector ★ │ ├── graphic/graphic_surface/ # Surface, BufferQueue, SyncFence ★ │ ├── bundlemanager/bundle_framework/ # BMS, installd, 包管理 ★ │ ├── communication/ipc/ # IPC 框架 ★ │ ├── distributedschedule/samgr/ # SA 管理器 ★ │ └── notification/common_event_service/ # CommonEvent 服务 ★ ├── base/ # 基础软件层 │ ├── startup/appspawn/ # App 进程孵化器 ★ │ ├── hiviewdfx/hilog/ # 日志框架 ★ │ └── security/access_token/ # 权限管理 ★ ├── commonlibrary/c_utils/ # C++ 公共库 ★ ├── vendor/hihope/dayu210/ # 厂商配置 └── build.sh # 编译入口脚本 ★ = 适配层所需模块

组件与 Inner Kits 机制

OH 的模块组织采用 子系统 → 组件 的两级结构，通过 bundle.json 定义元数据。组件通过 inner_kits 导出库和头文件，适配层的 JNI 代码链接这些 inner_kit 库来访问 OH 系统服务。

Inner Kit 引用格式："component_name:inner_kit_name"

  • oh_ability_manager_client.cpp → 链接 ability_runtime:ability_manager
  • oh_window_manager_client.cpp → 链接 window_manager:libwm
  • oh_bundle_mgr_client.cpp → 链接 bundle_framework:appexecfwk_core
  • oh_surface_bridge.cpp → 链接 graphic_2d:librender_service_client
  • oh_common_event_client.cpp → 链接 common_event_service:cesfwk_innerkits

3. AOSP 侧：App 运行时裁剪

3.1 客户端与服务端的天然分离

AOSP 14 的构建体系天然地将 App 运行时和系统服务分为两个独立的 JAR：

┌─────────────────────────────────────────────────────────────────────────┐ │ AOSP 14 构建目标 │ │ │ │ ┌─ 客户端 (App 进程内运行) ──────────────────────────────────────────┐ │ │ │ │ │ │ │ framework-minus-apex ──→ framework.jar │ │ │ │ │ │ │ │ 包含： │ │ │ │ • android.app.* (ActivityThread, Application, Intent, ...) │ │ │ │ • android.view.* (View, ViewGroup, Window, Surface, ...) │ │ │ │ • android.content.* (ContentProvider, ContentResolver, ...) │ │ │ │ • android.os.* (Binder, IBinder, Parcel, Handler, Looper, ...) │ │ │ │ • 所有 AIDL 生成的 Stub/Proxy 类 │ │ │ │ 不包含：任何 *Service 实现类 │ │ │ └────────────────────────────────────────────────────────────────────┘ │ │ │ │ ┌─ 服务端 (system_server) ── 本项目不需要 ───────────────────────────┐ │ │ │ services ──→ services.jar │ │ │ │ 包含：AMS, WMS, PMS, SystemServer.java 等 │ │ │ └────────────────────────────────────────────────────────────────────┘ │ └─────────────────────────────────────────────────────────────────────────┘
结论：AOSP 的构建体系已经为我们做了分离。编译 framework-minus-apex 即可获得仅含客户端代码的 framework.jar，无需任何裁剪工作。

6.8 为什么这么配置 — 设计决策说明

为什么 allow_sanitize_debug=true

dayu210 配置要求某些模块启用 CFI（控制流完整性）sanitizer，但测试模块未完全配置，导致 GN 阶段 assertion 失败。此参数跳过 sanitizer 检查，不影响产出的服务库功能。

为什么移除 sdk / ace_ets2bundle 依赖

这两个组件分别是 ArkTS SDK API 定义和 ETS 编译工具，仅用于构建 OH 应用（HAP），编译系统服务不需要。但 GN 构建系统的 ets2abc_config.gni 和 app_internal.gni 中无条件引用了它们，触发依赖解析失败。补丁将这些引用清空。

为什么 libjpeg-turbo 的 visibility 需要补丁

dayu210 板级 camera 模块（camera_pipeline_core）依赖 libjpeg-turbo 的 turbojpeg_static 目标，但该目标的 GN visibility 列表中未包含 camera 模块。这是 OH 6.1 dayu210 配置的一个遗漏，补丁添加了缺失的 visibility 条目。

为什么移除 ui_lite / graphic_utils_lite

ui_lite 是 OH lite 系统（小型设备）专用组件，其 updater 扩展依赖 graphic_utils_lite，在 standard 系统上不可用。由于 GN 解析整个依赖图，即使不编译此模块也会触发依赖检查失败。

为什么 musl syscall.h 需要 SYS_ 别名

musl 源码中有 #ifndef SYS_futex / #define SYS_futex SYS_futex_time64 的 fallback 逻辑。但 OH 构建系统生成的 bits/syscall.h 只包含 __NR_* 定义，没有对应的 SYS_* 别名。当 SYS_futex 未定义时，fallback 到不存在的 SYS_futex_time64，导致编译失败。补丁通过 sed 从 __NR_* 生成 SYS_* 别名。

为什么 dayu210_config.json 需要添加 Rust crates

OH 系统服务的依赖链中包含 Rust 编写的组件（如 rust_libc、rust_cfg_if 等），这些是间接依赖。GN 要求所有引用的组件都在产品配置中注册，否则 OHOS component not found 报错。

为什么构建隔离至关重要

构建隔离确保：(1) AOSP/OH 源码仓库始终可以 git pull 获取上游更新；(2) 编译过程可重复——任何人在任何时间应用补丁都能得到相同结果；(3) 补丁可以独立版本化管理、代码审查、回退。

为什么 Java 由 AOSP 编译但 Native 由 OH 编译

Java/DEX 字节码是平台无关的，AOSP Soong 编译的 framework.jar 可直接在 OH 上的 ART VM 中运行。但 Native .so 文件必须链接目标平台的 C 运行时——AOSP 编译产物链接 Bionic libc，无法被 OH 的 musl 动态链接器加载。因此 Native 层必须用 OH Clang（--target=aarch64-linux-ohos）交叉编译。

7.4 AOSP Native 层的交叉编译策略

可行性依据：ART 源码中已有大量 #if defined(__BIONIC__) ... #else ... 条件编译分支，交叉编译时 __BIONIC__ 未定义，自动走通用路径。

10. 结论与下一步

核心结论

  1. AOSP 构建体系天然支持客户端/服务端分离 — framework.jar（客户端）和 services.jar（服务端）是完全独立的构建目标，无需额外裁剪即可只编译客户端部分。
  2. OH 构建体系采用 GN + Ninja — 与 AOSP 的 Soong/Blueprint 完全不同。模块通过 bundle.json 声明元数据和依赖，inner_kits 机制实现组件间解耦。
  3. 两套构建系统通过文件系统交换产物，互不调用 — AOSP 产出 Java/DEX 层，OH 产出 Native 层和系统服务。交叉编译是连接两者的桥梁。
  4. AOSP 侧需要 9 个核心模块（Framework 客户端、ART、libcore、bionic、原生框架库、系统基础库、JNI 辅助库、预编译工具链、适配层），排除系统服务、系统应用和 HAL 层。
  5. OH 侧需要 9 个核心组件：ability_runtime、ability_base、window_manager、graphic_2d、graphic_surface、bundle_framework、common_event_service、ipc、samgr。其中 2 个需要打补丁。
  6. 补丁影响 2 个 OH 模块 9 个文件 + 4 个 AOSP 文件 + 5 个 OH 构建系统文件。所有修改以补丁形式管理，确保源码仓库始终干净。
  7. Java/DEX 产物可直接跨平台部署，AOSP Native 产物必须用 OH 工具链交叉重编（Layer 1-3 大部分已完成，10 个 .so 共约 14.6 MB）。

A.4 AIDL Stub 一次性生成

目标与原理

适配层的核心模式是 class inheritance：每个 Adapter 类继承 AIDL 自动生成的 IXxx.Stub 类。例如：

// ActivityManagerAdapter.java
public class ActivityManagerAdapter extends IActivityManager.Stub {
    // 必须覆写 IActivityManager.aidl 中定义的所有 249 个方法
    @Override
    public int startActivity(...) { ... }
    ...
}

IActivityManager.Stub 是 AIDL 编译器从 IActivityManager.aidl 自动生成的抽象类。我们需要：

  1. 用 AOSP 16 的 AIDL 编译器处理所有 .aidl 文件
  2. 生成 Java 源文件（IActivityManager.java 等）
  3. 将生成结果保存到适配层项目中
  4. 后续手动维护，不再依赖 AIDL 编译器自动生成

A.5 ART 虚拟机编译

ART 在本项目中的角色

ART（Android Runtime）是运行 Android App 的核心引擎。在 DAYU600 (OH) 上运行 Android App 必须部署 ART：

附录 B. appspawn-x 与 Zygote

B.1 appspawn-x 替代 Zygote 的设计

标准 Android 使用 Zygote 进程孵化 App 进程（frameworks/base/cmds/app_process/）。本项目使用 appspawn-x——一个混合 OH appspawn + Android Zygote 的进程孵化器，替代标准 Zygote：

功能标准 Android Zygoteappspawn-x
进程管理Zygote forkOH appspawn fork
安全策略Android SELinuxOH 沙箱 + Android 兼容
VM 创建AndroidRuntime::start()AppSpawnXRuntime (libart.so)
JNI 注册register_jni_procs()相同（通过 libandroid_runtime）
类预加载ZygoteInit.preload()AppSpawnXInit.preload()

B.2 appspawn-x 对 ART 的依赖

appspawn-x 的 BUILD.gn 引用以下 ART/Android 预编译库：

# 来自 framework/appspawn-x/BUILD.gn
deps = [
    "//third_party/android/runtime:libandroid_runtime",  # AndroidRuntime, JNI 注册
    "//third_party/android/runtime:libnativehelper",      # JniInvocation
    "//third_party/android/art:libart",                   # ART VM
    "//third_party/android/art:libdexfile",               # DEX 文件处理
]

这意味着：AOSP 编译 ART 后，需要将产物复制到 OH 源码树的 third_party/android/ 目录下，作为预编译库供 appspawn-x 链接。

B.3 appspawn-x 编译（OH GN 构建）
