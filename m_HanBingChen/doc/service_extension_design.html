1. 概述

1.1 背景与目标

Android Service 是 Android 四大组件之一，用于在后台执行长时间运行的操作或为其他应用提供跨进程服务。在 OpenHarmony（OH）系统中，对应的组件是 ServiceExtensionAbility（Stage 模型，ExtensionAbilityType = SERVICE = 3）。

本文档设计 Android Service 与 OH ServiceExtensionAbility 之间的完整适配层，使得：

设计目标

  • Android App 无感运行：Android 应用调用 startService() / bindService() / stopService() 等标准 API，适配层透明地将请求转发到 OH ServiceExtensionAbility
  • 跨系统互调：支持 Android App 调用 OH 原生 ServiceExtension，也支持 OH App 调用 Android Service
  • 连接管理：完善 bindService/unbindService 的连接生命周期管理，支持多并发连接
  • 修复现有缺陷：修复当前代码中已识别的6个问题点

1.2 三种适配场景

S1
Android App → Android Service
（双端运行于OH）
S2
Android App → OH ServiceExtension
（跨系统正向调用）
S3
OH App → Android Service
（跨系统反向桥接）
场景 S1：最常见场景。Android 应用 A 启动/绑定同样运行在 OH 上的 Android Service B。适配层在 IPC 两端均需介入：正向通过 ActivityManagerAdapter 将 AMS 调用路由到 OH IAbilityManager，反向通过 AbilitySchedulerBridge 将 OH 回调转换为 IApplicationThread 调度。
场景 S2：Android 应用调用 OH 原生 ServiceExtensionAbility。正向路径与 S1 一致，但反向无需 AbilitySchedulerBridge（OH Service 自行处理 IAbilityScheduler 回调）。连接回调通过 IAbilityConnection → IServiceConnection 桥接传递 remoteBinder。
场景 S3：OH 原生应用通过 ConnectAbility 连接 Android Service。反向桥接层接收 ScheduleConnectAbility，需触发 Android Service 完整创建流程（scheduleCreateService + scheduleBindService），并将 Service.onBind() 返回的 IBinder 回传给 OH 调用方。

2. Android Service 机制分析

2.1 核心架构

Android Service 运行在应用进程内，但其生命周期由系统服务 ActivityManagerService (AMS) 管理。Service 有两种运行模式：

模式启动方式生命周期特点
Started Service startService(Intent) onCreate → onStartCommand → 运行 → onDestroy 独立运行，需调用 stopSelf() 或 stopService() 终止
Bound Service bindService(Intent, conn, flags) onCreate → onBind → 运行 → onUnbind → onDestroy 基于连接计数，所有客户端 unbind 后自动销毁
混合模式 startService + bindService onCreate → onStartCommand + onBind → 运行 → onUnbind → onDestroy 需同时满足 stop + 所有 unbind 才销毁

3. OH ServiceExtensionAbility 机制分析

3.1 Stage 模型下的 ServiceExtension

在 OpenHarmony Stage 模型中，ServiceExtensionAbility 是后台服务组件，对应 ExtensionAbilityType::SERVICE = 3。与 Android Service 的核心区别在于：

特性Android ServiceOH ServiceExtensionAbility
启动方式startService + bindService（两种模式）ConnectAbility（类bind） + StartAbility（类start，但不常用于Service）
连接模型IServiceConnection 单回调IAbilityConnection 双回调（ConnectDone + DisconnectDone）
服务发布publishService（显式发布 IBinder）OnConnect 返回值自动传递 remoteObject
实例管理单实例，多客户端共享单实例，多客户端共享
前台服务setServiceForeground + Notification无直接等价（通过 WantAgent 实现通知）
隔离进程支持 isolatedProcess不支持
onStartCommand返回重启策略（START_STICKY等）OnCommand 无返回值

4. 适配方案设计

4.1 场景 S1：Android App → Android Service（双端运行于OH）

这是最常见的场景。Android 应用的 startService / bindService 调用通过适配层路由到 OH 系统服务，OH 系统服务再调度目标 Android Service 的生命周期。

正向路径（App → 系统）

  1. Android App 调用 ContextImpl.startService(intent)
  2. Framework 层调用 IActivityManager.startService()
  3. ActivityManagerAdapter.startService() 拦截，将 Intent 转换为 Want
  4. 通过 JNI 调用 nativeStartAbility(want)，映射到 OH IAbilityManager::StartAbility(want)
  5. OH AbilityManagerService 启动目标 ServiceExtensionAbility

反向路径（系统 → Service App）

  1. OH 系统通过 IAbilityScheduler.ScheduleCommandAbility(want) 通知目标进程
  2. C++ AbilitySchedulerAdapter 接收调用，通过 JNI 回调到 Java 层
  3. AbilitySchedulerBridge.onScheduleCommandAbility() 转换参数
  4. 调用 IApplicationThread.scheduleServiceArgs(token, startId, flags, intent)
  5. Android Service 的 onStartCommand() 被正常触发
关键点：场景 S1 中，目标 Android Service 也运行在 OH 环境中，其进程的 IAbilityScheduler 由 AbilitySchedulerAdapter（C++ Stub）实现，负责接收 OH 系统的调度并转发给 Android Framework 的 IApplicationThread。

4.2 场景 S2：Android App → OH ServiceExtension

Android 应用调用 OH 原生的 ServiceExtensionAbility。正向路径与 S1 一致，区别在于反向路径——OH ServiceExtension 自行处理 IAbilityScheduler 回调，无需 AbilitySchedulerBridge 介入。

连接回调路径

  1. Android App 调用 bindService(intent, serviceConnection, flags)
  2. ActivityManagerAdapter.bindService() 拦截：
    • 将 IServiceConnection 注册到 ServiceConnectionRegistry，获得 connectionId
    • 通过 JNI 调用 nativeConnectAbility(want, connectionId)
  3. C++ 侧为该 connectionId 创建一个 AbilityConnectionAdapter 实例并传给 OH ConnectAbility
  4. OH ServiceExtension 的 OnConnect() 返回 remoteObject
  5. OH 系统调用 IAbilityConnection.OnAbilityConnectDone(element, remoteObject, resultCode)
  6. C++ AbilityConnectionAdapter 通过 JNI 回调到 Java
  7. AbilityConnectionBridge.onAbilityConnectDone() 查找 ServiceConnectionRegistry，调用 IServiceConnection.connected(componentName, binder, false)

4.3 场景 S3：OH App → Android Service（反向桥接）

OH 原生应用通过 ConnectAbility 连接到 Android Service。这是最复杂的场景，需要完整模拟 Android Service 的创建和绑定流程。

完整流程

  1. OH App 调用 ConnectAbility(want, abilityConnection)，目标指向 Android Service
  2. OH AbilityManagerService 识别目标为 Android 兼容 Ability，调度到 Android 兼容进程
  3. AbilitySchedulerAdapter（C++ Stub）接收 ScheduleConnectAbility(want)
  4. 通过 JNI 回调到 AbilitySchedulerBridge.onScheduleConnectAbility(want)
  5. 桥接层执行以下 Android Service 创建序列：
    • 如果 Service 未创建：调用 IApplicationThread.scheduleCreateService(token, serviceInfo)
    • 调用 IApplicationThread.scheduleBindService(token, intent, rebind=false)
  6. Android Service 的 onBind(intent) 返回 IBinder
  7. Service 调用 AMS.publishService(token, intent, binder)
  8. ActivityManagerAdapter.publishService() 拦截，通过 JNI 调用 nativePublishService(connectionId, binder)
  9. C++ 侧将 IBinder 包装后通过 ScheduleConnectAbilityDone 回传给 OH
  10. OH 系统调用原始调用方的 IAbilityConnection.OnAbilityConnectDone()
注意：场景 S3 中，AbilitySchedulerBridge 需要管理 Android Service 的 token，并维护一个 Service 实例追踪表，确保同一 Service 不会被重复创建。同时需要处理 Android IBinder 到 OH IRemoteObject 的跨系统 Binder 转换。

4.4 ServiceConnectionRegistry 设计

新增单例类 ServiceConnectionRegistry，统一管理 bindService/unbindService 的连接生命周期。

类设计

/**
 * Singleton registry for tracking Android Service connections mapped to OH AbilityConnections.
 *
 * Thread-safe: all public methods are synchronized.
 */
public class ServiceConnectionRegistry {
    private static final ServiceConnectionRegistry sInstance = new ServiceConnectionRegistry();

    // Connection tracking
    private final Map<IBinder, ConnectionRecord> mConnectionsByToken = new HashMap<>();
    private final Map<Integer, ConnectionRecord> mConnectionsById = new HashMap<>();
    private int mNextConnectionId = 1;

    static class ConnectionRecord {
        final int connectionId;
        final IServiceConnection connection;
        final IBinder connectionToken;  // IServiceConnection.asBinder()
        ComponentName componentName;
        boolean isBound;

        ConnectionRecord(int id, IServiceConnection conn, IBinder token) { ... }
    }

    public static ServiceConnectionRegistry getInstance() { return sInstance; }

    /**
     * Register a new connection. Called from ActivityManagerAdapter.bindService().
     * @return connectionId for passing to native ConnectAbility
     */
    public synchronized int register(IServiceConnection connection, ComponentName name) { ... }

    /**
     * Unregister a connection. Called from ActivityManagerAdapter.unbindService().
     * @return connectionId for passing to native DisconnectAbility, or -1 if not found
     */
    public synchronized int unregister(IServiceConnection connection) { ... }

    /**
     * Called when OH IAbilityConnection.OnAbilityConnectDone fires.
     * Dispatches to the corresponding IServiceConnection.connected().
     */
    public synchronized void onConnected(int connectionId, String bundleName,
            String abilityName, IBinder remoteBinder) { ... }

    /**
     * Called when OH IAbilityConnection.OnAbilityDisconnectDone fires.
     * Dispatches IServiceConnection.connected(name, null, dead=true).
     */
    public synchronized void onDisconnected(int connectionId) { ... }

    /**
     * Find connectionId by IServiceConnection token.
     */
    public synchronized int findConnectionId(IBinder connectionToken) { ... }
}
设计决策：使用 connectionId（int）而非直接传递 IBinder 到 native 层，原因如下：
  • OH 的 IAbilityConnection 是 C++ 对象，无法直接持有 Java IBinder 引用
  • connectionId 作为轻量级标识符可安全跨 JNI 边界传递
  • Java 侧通过 connectionId 查表即可恢复完整的 IServiceConnection 引用

7. 关键差异与解决方案

#差异点Android 行为OH 行为影响解决方案
1 启动与绑定分离 startService 和 bindService 是独立操作，Service 可以同时被 start 和 bind ConnectAbility（bind）和 StartAbility + ScheduleCommandAbility（start）相对独立 中 适配层维护 Service 的混合模式状态。startService 使用 StartAbility，bindService 使用 ConnectAbility，两者可并存。销毁条件需在适配层判断
2 连接回调差异 IServiceConnection.connected(name, binder, dead) — 单个方法，通过 dead 参数区分连接/断开 IAbilityConnection 有两个独立方法：OnAbilityConnectDone 和 OnAbilityDisconnectDone 低 ConnectDone 映射为 connected(dead=false)，DisconnectDone 映射为 connected(dead=true)
3 服务发布机制 Service.onBind() 返回 IBinder 后，需显式调用 AMS.publishService() 将 Binder 发布给客户端 OnConnect() 直接返回 remoteObject，系统自动传递给调用方 高 场景 S1/S3：publishService 拦截 IBinder，通过 JNI 传递给等待的 C++ AbilityConnectionAdapter，完成 ConnectAbilityDone 回调。场景 S2：OH 原生 Service 自动处理，无需适配
4 onStartCommand 返回值 返回 START_STICKY / START_NOT_STICKY / START_REDELIVER_INTENT，控制 Service 被杀后的重启行为 OnCommand 无返回值，系统默认管理 Service 重启 中 适配层记录 onStartCommand 返回值但不传递给 OH。OH 系统自行管理进程重启策略
5 前台服务 setServiceForeground() + Notification 对象 无直接等价 API 中 Phase 1 以 stub 处理。后续可通过 OH WantAgent + Notification 实现类似效果
6 Rebind 机制 Service.onUnbind() 返回 true 时，下次 bind 触发 onRebind() 而非 onBind() OH 每次 ConnectAbility 都调用 OnConnect，无 rebind 概念 低 适配层在 unbindFinished 中记录 doRebind 标志。当同一 Service 再次收到 ScheduleConnectAbility 时，如 doRebind=true 则调用 scheduleBindService(rebind=true)
7 连接多路复用 多个客户端可分别 bind 同一 Service，每个客户端有独立的 IServiceConnection 多个客户端可分别 ConnectAbility，每个有独立的 IAbilityConnection 低 ServiceConnectionRegistry 按 connectionId 管理，天然支持多连接
8 IBinder 跨系统传递 IBinder 是 Android Binder 框架的核心 IRemoteObject 是 OH IPC 框架的核心 高 需要实现 IBinder ↔ IRemoteObject 的双向包装。C++ 侧通过 BinderBridge 将 Android Binder transaction 转换为 OH IPC transaction

9. 现有代码缺陷修复

通过代码审查，发现以下6个影响 Service 适配正常工作的缺陷：

缺陷 #1：无连接跟踪机制

位置ActivityManagerAdapter.java — bindService()
现状bindService() 接收 IServiceConnection 参数但从未保存。nativeConnectAbility() 始终传递 connectionId = 0，导致 OH 侧连接回调无法关联回正确的 IServiceConnection
影响高 — bindService 永远无法正确回调客户端
修复引入 ServiceConnectionRegistry，在 bindService 中注册 IServiceConnection 并获取唯一 connectionId，传递给 native 层

缺陷 #2：全局单一 AbilityConnection 实例

位置oh_callback_handler.h/cpp — abilityConnection_ 字段
现状OHCallbackHandler 仅持有单个 abilityConnection_ 实例。当多个 bindService 并发调用时，新的连接会覆盖旧的，导致之前的连接回调丢失
影响高 — 仅支持同时存在一个 Service 连接
修复将 abilityConnection_ 替换为 std::unordered_map<int, sptr<AbilityConnectionAdapter>> connectionMap_，以 connectionId 为键管理多个并发连接

缺陷 #3：stopService 未执行 OH IPC 调用

位置ActivityManagerAdapter.java — stopService()
现状方法体仅 logStub("stopService") 并 return 1，未调用任何 OH 系统服务。被 stop 的 Service 实际上仍在 OH 侧运行
影响高 — Service 无法被停止，造成资源泄漏
修复实现 Intent 到 Want 转换，调用 nativeStopServiceAbility(want)，映射到 OH IAbilityManager::StopServiceAbility(want)

缺陷 #4：publishService / unbindFinished 为空 Stub

位置ActivityManagerAdapter.java — publishService() 和 unbindFinished()
现状两个方法均为空实现。publishService 不处理意味着 Service.onBind() 返回的 IBinder 永远无法到达客户端。unbindFinished 不处理意味着 rebind 信息丢失
影响高 — Bound Service 的核心功能完全不可用
修复
  • publishService：实现 nativePublishService(connectionId, binder)，将 IBinder 传递给 C++ 侧等待的 AbilityConnectionAdapter
  • unbindFinished：记录 doRebind 标志到 Service 状态表，供后续 ScheduleConnectAbility 判断是否使用 rebind 路径

缺陷 #5：AbilitySchedulerBridge Service 回调未实现

位置AbilitySchedulerBridge.java — onScheduleConnectAbility(), onScheduleDisconnectAbility(), onScheduleCommandAbility()
现状三个方法均仅打印日志，不执行任何实际的 IApplicationThread 调用。这意味着 OH 系统的 Service 调度请求被完全忽略
影响高 — 场景 S1、S3 的 Service 生命周期回调完全不工作
修复
  • onScheduleConnectAbility：通过反射或直接调用 IApplicationThread.scheduleCreateService()（如 Service 未创建）+ scheduleBindService()
  • onScheduleDisconnectAbility：调用 IApplicationThread.scheduleUnbindService()
  • onScheduleCommandAbility：调用 IApplicationThread.scheduleServiceArgs()，将 OH Want 转换回 Android Intent，传递 startId

缺陷 #6：serviceDoneExecuting 为空实现

位置ActivityManagerAdapter.java — serviceDoneExecuting()
现状空 stub。Android Service 生命周期方法执行完毕后会调用此方法通知 AMS。在 OH 环境中虽无直接等价物，但缺少此通知可能影响服务状态追踪
影响低 — OH 不依赖此通知管理 Service 状态
修复保持 stub，但增加适配层内部状态更新逻辑（如更新 ServiceConnectionRegistry 中的 Service 状态）
