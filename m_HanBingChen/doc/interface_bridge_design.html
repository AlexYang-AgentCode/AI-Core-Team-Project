1. 概述

本文档描述了 Android 到 OpenHarmony（OH）接口桥接层的详细设计。桥接层拦截 Android 应用对四个关键 SystemServer 接口的 Binder IPC 调用，并将其路由到等效的 OpenHarmony 系统服务接口。

设计目标

  • 完整 API 覆盖：4 个 Android 接口中的每个方法都被显式处理——要么桥接到 OH 等效接口，要么部分映射，要么以安全默认返回值进行存根处理。
  • 非侵入式：使用 Java 动态代理拦截调用，无需修改 Android 框架源代码或 AIDL 定义。
  • Android API 兼容性：应用继续使用标准 Android API；桥接层对应用代码完全透明。
  • 分类映射：方法按功能类别组织，并带有清晰的映射状态标签。

2.2 适配安装机制

采用类继承方式（extends IXxx.Stub）替代动态代理。安装方式有两种：

方式一：运行时替换（当前实现）

ServiceInterceptor 通过反射将 Adapter 实例设入框架的单例字段：

  1. 创建 Adapter 实例（如 new ActivityManagerAdapter(adaptationLayer)）
  2. 通过反射定位 Singleton<T> 的 mInstance 字段
  3. 直接将 Adapter 实例设入 mInstance（无需动态代理）

方式二：AOSP 源码修改（已实现，推荐生产方式）

直接修改 Android 框架源码，在 Singleton<T>.create() 和初始化方法中增加OH环境检测。 修改后的AOSP源码位于 aosp_patches/ 目录，同时原始AOSP文件已直接修改。

修改模式（条件式替换，保留回退）：

// 示例：ActivityManager.java 中修改后的 create() 方法
private static final Singleton<IActivityManager> IActivityManagerSingleton =
    new Singleton<IActivityManager>() {
        @Override
        protected IActivityManager create() {
            // [OH_ADAPTER] 检测是否运行在OH环境
            if (OHEnvironment.isOHEnvironment()) {
                try {
                    return new ActivityManagerAdapter();
                } catch (Exception e) {
                    Log.w("ActivityManager",
                        "OH adapter init failed, falling back to original", e);
                }
            }
            // 原始AOSP实现（回退路径）
            final IBinder b = ServiceManager.getService(Context.ACTIVITY_SERVICE);
            final IActivityManager am = IActivityManager.Stub.asInterface(b);
            return am;
        }
    };

OH环境检测机制：

// OHEnvironment.java
public static boolean isOHEnvironment() {
    // 1. 检查系统属性 persist.oh.adapter.enabled
    // 2. 回退：通过native方法检测OH IPC框架是否可用
    return nativeIsOHEnvironment();
}

设计优势：修改后的AOSP代码在非OH设备上完全兼容原始行为（isOHEnvironment()返回false），无需条件编译。

所有 4 个 Adapter 类都采用相同的类继承模式（extends IXxx.Stub）：

public class ActivityManagerAdapter extends IActivityManager.Stub {
    private final long mOhAbilityManager;    // OH 服务句柄

    public ActivityManagerAdapter() {
        mOhAbilityManager = OHEnvironment.nativeGetOHAbilityManagerService();

    // [BRIDGED] 每个 AIDL 方法都有精确类型签名的 @Override
    @Override

            String callingPackage, String callingFeatureId,
            Intent intent, ...) {
        IntentWantConverter.WantParams want =
            IntentWantConverter.intentToWant(intent);

    @Override

}

关键实现：handleStartActivity

handleStartActivity 方法展示了 Activity 启动的核心桥接逻辑：

private Object handleStartActivity(String methodName, Object[] args) {
    Intent intent = findInArgs(args, Intent.class);
    if (intent == null) {
        Intent[] intents = findInArgs(args, Intent[].class);
        if (intents != null && intents.length > 0) intent = intents[0];
    }
    if (intent == null) return 0;

    // 将 Android Intent 转换为 OH Want 参数
    IntentWantConverter.WantParams want = IntentWantConverter.intentToWant(intent);

    // 通过 JNI 调用 OH AbilityManager
    return mAdapter.nativeStartAbility(
        want.bundleName, want.abilityName,
        want.action, want.uri, want.extrasJson);
}

9. 调用流程示例

示例 1：startActivity

App: context.startActivity(intent)
  -> ActivityTaskManager.getService().startActivity(...)
    -> [Proxy intercept] IActivityTaskManagerBridge.invoke()
      -> handleStartActivity()
        -> IntentWantConverter.intentToWant(intent)
           {bundleName: "com.example.target",
            abilityName: "MainAbility",
            action: "ohos.want.action.viewData",
            uri: "...", extrasJson: "{...}"}
        -> OHEnvironment.nativeStartAbility(want)
          -> [JNI] adapter_bridge.cpp
            -> [OH IPC] IAbilityManager::StartAbility(Want)
              -> OH AbilityManagerService
                -> Target Ability launched

示例 2：窗口重新布局

ViewRootImpl: mWindowSession.relayout(window, attrs, ...)
  -> [Proxy intercept] IWindowSessionBridge.invoke()
    -> case "relayout":
      -> [BRIDGED] -> OH ISession.UpdateSessionRect
      -> Phase 1: method.invoke(mOriginal, args) // passthrough
      -> Future: OH IPC to ISession for actual layout update

  -> WindowManagerGlobal.getWindowManagerService()

12. Limitations and Future Work

当前限制（第一阶段）

  • 日志模式：大部分 BRIDGED 方法当前仅记录日志并返回默认值。OH IPC 调用尚未连接到真实的 OH 系统服务。
  • 存根方法：约 311 个方法返回安全默认值。部分方法（广播、ContentProvider、权限）需要完全不同的 OH 模式，无法直接桥接。
  • 反向回调：OH 到 Android 的回调（IAppScheduler、IAbilityScheduler、ISessionStage）已设计但尚未完全接通。
  • 单进程：当前适配仅在应用进程内工作。SystemServer 侧的桥接尚未实现。

后续工作

  • 第二阶段：将日志模式替换为实际的 OH IPC 客户端调用，实现所有 BRIDGED 方法的真实桥接。
  • 第三阶段：实现反向回调桥接（OH 生命周期事件 → Android 生命周期回调）。
  • 第四阶段：处理广播 → CommonEventManager 和 ContentProvider → DataShareExtensionAbility 的模式适配。
  • 第五阶段：SystemServer 侧桥接，处理服务端发起的操作。
  • 测试：自动化测试套件，验证每个适配方法返回正确类型并处理边界情况。
