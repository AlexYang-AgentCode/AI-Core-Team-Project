一、项目背景与目标

本报告旨在评估将 Android Framework 与 OpenHarmony 系统服务进行对接的技术可行性。核心构想是在 Android Framework 中增加一个适配层（Adaptation Layer），实现以下两个目标：

  1. App → 系统服务方向：将 Android App 作为 Client 调用系统服务的接口，转换为调用 OpenHarmony 系统服务的接口；
  2. 系统服务 → App 方向：将 OpenHarmony 系统服务调用 App 的接口，转换为 Android App 能够理解的接口。

两个系统均采用 Binder 机制进行跨进程 IPC 通信，但其接口定义、数据结构和底层实现存在显著差异。

三、核心接口语义映射分析

3.1 应用生命周期管理

功能AndroidOpenHarmony映射难度
应用附着IActivityManager.attachApplication()IAppMgr.AttachApplication()中 参数结构不同但语义相近
启动Activity/AbilityIActivityTaskManager.startActivity(Intent)IAbilityManager.StartAbility(Want)高 Intent与Want模型差异大
前台/后台切换通知IApplicationThread.scheduleTransaction()IAppScheduler.ScheduleForegroundApplication()中 Android用ClientTransaction封装
生命周期回调scheduleTransaction(LaunchActivityItem...)IAbilityScheduler.ScheduleAbilityTransaction()高 生命周期状态机不同
Service绑定IActivityManager.bindService()IAbilityManager.ConnectAbility()中 语义类似，连接回调机制不同
内存通知IApplicationThread.scheduleTrimMemory()IAppScheduler.ScheduleMemoryLevel()低 可直接映射

3.2 窗口管理

功能AndroidOpenHarmony映射难度
创建窗口IWindowSession.addToDisplay()ISceneSessionManager.CreateAndConnectSpecificSession()（SA 4607，返回 ISession 每窗口代理）极高 Surface/图形系统完全不同
窗口布局IWindowSession.relayout()ISession.UpdateSessionRect()（每窗口代理方法）高 布局参数、Insets模型不同
焦点管理Android IWindowManager 内部IWindowManagerAgent.UpdateFocusChangeInfo()中
输入事件分发通过 InputChannel通过 Session 机制极高 底层通道机制完全不同

3.3 生命周期状态机差异

Android Activity:
  onCreate → onStart → onResume → onPause → onStop → onDestroy

OpenHarmony Ability (Stage模型):
  onCreate → onWindowStageCreate → onForeground → onBackground → onWindowStageDestroy → onDestroy

关键差异：OpenHarmony 的 Ability 生命周期与 WindowStage 紧耦合，而 Android 的 Activity 生命周期与 Window 相对松耦合。适配层需要在两种状态机之间建立正确的映射关系。

六、核心技术挑战

挑战1：Binder 跨系统互通

  • Android Binder 和 OH IPC 框架是两套独立的内核驱动和用户态框架
  • Android 使用 AIDL 生成 Java Proxy/Stub；OH 使用 C++ IRemoteProxy/IRemoteStub
  • 必须解决：两个 Binder 域之间的消息转发机制
  • 方案：在 App 进程内通过 JNI 桥接，Java 层拦截 Android Binder 调用，转入 C++ 层使用 OH IPC 框架发起跨进程调用

挑战2：Intent ↔ Want 转换

  • Android Intent (Java)：action / category / data(URI) / extras(Bundle) / component
  • OH Want (C++)：bundleName / abilityName / action / entities / parameters
  • 需要建立完整的映射规则和转换层，尤其是隐式 Intent 的解析

挑战3：图形渲染管道

  • Android：SurfaceFlinger + BufferQueue + HWC
  • OH：RenderService + Surface + HWC
  • 这是最大的难点，窗口合成和显示是底层紧耦合的
  • 需要共享 GraphicBuffer 或建立 Surface 代理层

挑战4：输入系统

  • Android：InputManagerService → InputChannel (socketpair) → App
  • OH：多模输入子系统 → Session → App
  • 输入事件格式（MotionEvent vs. PointerEvent）和分发路径完全不同

挑战5：运行环境共存

  • Android App 进程运行在 ART (Android Runtime) 上
  • OH IPC 客户端库是 C++ 实现
  • 需要确保同一进程中可同时加载 ART 和 OH IPC 动态库

八、适配层核心模块设计

模块1：LifecycleAdapter（生命周期适配）

App → 系统服务方向

ActivityManager.startActivity(Intent)
  → LifecycleAdapter.convertIntent(Intent) → Want
  → [JNI Bridge]
  → AbilityManagerProxy.StartAbility(Want)

系统服务 → App 方向

IAppScheduler.ScheduleLaunchAbility()
  → [JNI Bridge]
  → LifecycleAdapter.convertToClientTransaction()
  → IApplicationThread.scheduleTransaction(ClientTransaction)

模块2：WindowAdapter（窗口适配）

App → 系统服务方向

ViewRootImpl → IWindowSession.addToDisplay()
  → WindowSessionAdapter.addToDisplay()
  → [JNI Bridge]
  → ISceneSessionManager.CreateAndConnectSpecificSession() (SA 4607)
  → 返回 ISession 每窗口代理（存储供后续使用）

ViewRootImpl → IWindowSession.relayout()
  → WindowSessionAdapter.relayout()
  → [JNI Bridge]
  → ISession.UpdateSessionRect()  (每窗口代理，非 ISceneSessionManager)

系统服务 → App 方向

ISessionStage.UpdateRect()
  → [JNI Bridge] → SessionStageBridge
  → IWindow.resized()

模块3：IPC Bridge（跨 Binder 域桥接）

核心设计思路：

  1. 拦截点：在 Android Framework 的 ServiceManager.getService() 处拦截，返回适配层的 Proxy 替代原 SystemServer 的 Binder
  2. JNI 层：Java Proxy 通过 JNI 调用 C++ 适配代码
  3. OH IPC 调用：C++ 层使用 OH 的 IRemoteProxy 发起真正的跨进程调用
  4. 反向回调：C++ 层实现 OH 的 IRemoteStub 接收系统服务回调，通过 JNI 调回 Java 层

九、可行性评估

维度评估说明
基本生命周期可行startAbility、前后台切换、销毁等核心流程可映射
Service 绑定可行bindService ↔ ConnectAbility 语义相近
简单 UI 渲染有条件可行需要桥接图形 Surface，工作量极大
输入事件有条件可行需要转换输入事件格式和分发通道
多窗口/分屏极难两套窗口模型差异太大
ContentProvider / DataAbility有条件可行数据访问模式不同但可桥接
通知/广播需额外适配OH 无广播机制，需用 CommonEvent 替代
权限模型需映射两套权限系统不同

十、建议实施路径

阶段一：最小闭环验证（POC）

  1. 实现 IPC Bridge 基础框架（JNI 桥接 Android Binder → OH IPC）
  2. 打通 startActivity(Intent) → 适配层 → OH AbilityManager.StartAbility(Want) 一条链路
  3. 实现反向回调：OH 通知 App 进入前台

阶段二：生命周期完整覆盖

  1. 完成完整的 Activity/Ability 生命周期映射
  2. 实现 Service/ServiceExtension 绑定
  3. 实现 attachApplication 流程

阶段三：窗口系统（最难）

  1. 桥接 Surface 系统（可能需要修改 OH RenderService 或引入共享内存方案）
  2. 桥接输入事件分发
  3. 实现窗口创建、布局、销毁完整流程

十一、结论

技术上有条件可行，但工程量巨大。核心判断如下：

  1. 生命周期和服务管理层的适配是可行的。两套系统的概念模型足够相似（Activity ↔ Ability, Service ↔ ServiceExtension, Intent ↔ Want），可以建立合理的映射关系。
  2. 图形/窗口系统是最大瓶颈。Surface 和输入通道是底层紧耦合的，需要 kernel 层面的协调或图形缓冲区共享机制。
  3. 前提条件：Android App 进程需要能同时加载 Android Runtime (ART) 和 OH IPC 动态库，即运行环境需要支持两套框架共存。这要求底层 Linux 内核同时加载两个 Binder 驱动（或 OH 使用用户态 IPC 方案）。
  4. 建议：先用阶段一做最小 POC 验证，打通一条完整的调用链路（startActivity → Ability 启动 → 前台回调），再决定是否全面推进。

该方案的本质类似于 Android 兼容层（如 Windows Subsystem for Android 的逆向思路），在工业界有类似先例，但需要大量底层系统级开发工作。
