1. 分析背景与目标

1.1 分析背景

Android-OH 适配项目的核心架构是：在 OH 系统上运行 Android App，保留 Android Framework 客户端代码和 ART 虚拟机，但将系统服务替换为 OH 实现。因此需要编译一个仅包含 App 运行时的 AOSP 子集，不包含 system_server 及其承载的所有系统服务。

本报告对 AOSP 14（android-14.0.0_r1）的源码结构和构建体系进行分析，明确哪些模块需要编译、哪些可以排除，并给出构建策略建议。

1.2 裁剪目标

需要保留（App 进程内运行）

  • ART 虚拟机 — Dex 字节码加载与执行、JIT/AOT 编译、GC、JNI
  • Framework 客户端代码 — ActivityThread、View 体系、Intent、所有 AIDL 生成的 Stub/Proxy 类
  • Framework 原生库 — libandroid_runtime（JNI 桥接）、Binder 客户端、Surface 客户端
  • Java 核心库 — java.lang.*、java.util.*、java.io.* 等
  • C 运行时 — libc、libm、动态链接器
  • 系统资源 — framework-res.apk

需要排除（在 OH 系统中由 OH 服务替代）

  • system_server 进程及所有系统服务（AMS、WMS、PMS 等）
  • 系统应用（SystemUI、Launcher、Settings 等）
  • HAL 服务层

2. AOSP 14 构建体系分析

2.1 AOSP 源码顶层结构

服务器上的 AOSP 14 源码位于 /root/aosp/，包含以下核心目录：

/root/aosp/ ├── art/ # ART 虚拟机运行时 ├── bionic/ # Android C 运行时 (libc, libm, linker) ├── build/ # 构建系统 (Soong/Blueprint) ├── external/ # 第三方依赖库 ├── frameworks/ # Android Framework │ ├── base/ # 核心框架 (Java + JNI + 资源) │ └── native/ # 原生框架库 (Binder, Surface, Input) ├── libcore/ # Java 核心库 (java.lang.*, java.util.*) ├── libnativehelper/ # JNI 辅助库 ├── prebuilts/ # 预编译工具链 (Clang, JDK, SDK tools) └── system/ # 系统基础库 (libbase, liblog, libutils)

2.2 Framework 构建目标层次

AOSP 14 的 Framework 构建在 frameworks/base/Android.bp 中定义了清晰的层次结构：

层次构建目标类型说明
1 framework-core-sources filegroup 核心源码集合：core/java/ 下所有 *.java + *.aidl 文件，包含 android.app.*、android.view.*、android.content.*、android.os.* 等所有客户端 API
2 framework-non-updatable-sources filegroup 聚合 framework-core-sources + 领域源码（graphics、telephony、media、wifi、location、opengl 等）+ 原生/HAL AIDL 接口
3 framework-minus-apex-defaults java_defaults 编译配置：编译 framework-non-updatable-sources，启用 AIDL 处理（generate_get_transaction_name: true）
4 framework-minus-apex java_library
(installable)		 实际的 framework.jar，安装到设备。使用 stem: "framework"。这就是客户端 Framework——不含任何系统服务代码
5 framework java_library
(not installable)		 编译时依赖目标：framework-minus-apex + framework-updatable-stubs。仅用于其他模块编译时引用
关键发现：framework-minus-apex 即为最终产物 framework.jar，它仅包含客户端代码。系统服务（services.jar）是完全独立的构建目标，定义在 frameworks/base/services/Android.bp 中。两者之间没有编译依赖关系。

2.3 客户端与服务端的天然分离

AOSP 14 的构建体系天然地将 App 运行时和系统服务分为两个独立的 JAR：

┌─────────────────────────────────────────────────────────────────────────┐ │ AOSP 14 构建目标 │ │ │ │ ┌─ 客户端 (App 进程内运行) ──────────────────────────────────────────┐ │ │ │ │ │ │ │ framework-minus-apex ──→ framework.jar │ │ │ │ │ │ │ │ 包含： │ │ │ │ • android.app.* (ActivityThread, Application, Intent, ...) │ │ │ │ • android.view.* (View, ViewGroup, Window, Surface, ...) │ │ │ │ • android.content.* (ContentProvider, ContentResolver, ...) │ │ │ │ • android.os.* (Binder, IBinder, Parcel, Handler, Looper, ...) │ │ │ │ • 所有 AIDL 生成的 Stub/Proxy 类 │ │ │ │ • com.android.internal.* (内部工具类) │ │ │ │ │ │ │ │ 不包含：任何 *Service 实现类 │ │ │ │ │ │ │ └────────────────────────────────────────────────────────────────────┘ │ │ │ │ ┌─ 服务端 (system_server 进程内运行) ── 本项目不需要 ────────────────┐ │ │ │ │ │ │ │ services ──→ services.jar │ │ │ │ │ │ │ │ 包含： │ │ │ │ • ActivityManagerService, WindowManagerService │ │ │ │ • PackageManagerService, PowerManagerService │ │ │ │ • SystemServer.java (入口) │ │ │ │ • services.core, services.accessibility, services.autofill, ... │ │ │ │ │ │ │ └────────────────────────────────────────────────────────────────────┘ │ └─────────────────────────────────────────────────────────────────────────┘
结论：AOSP 的构建体系已经为我们做了分离。编译 framework-minus-apex 即可获得仅含客户端代码的 framework.jar，无需任何裁剪工作。services 目标完全不需要编译。

5. 排除模块详细分析

5.1 系统服务（frameworks/base/services/）

services/ 目录包含 system_server 进程中运行的所有系统服务。在适配项目中，这些服务全部由 OH 系统服务替代：

Android 服务services/ 子目录OH 替代
ActivityManagerServiceservices/core/AbilityManagerService (SA 180)
WindowManagerServiceservices/core/SceneSessionManager (SA 4607)
PackageManagerServiceservices/core/BundleManagerService (SA 401)
PowerManagerServiceservices/core/PowerMgrService
InputManagerServiceservices/core/MultimodalInput
SurfaceFlingerframeworks/native/services/surfaceflinger/RenderService
SystemServer.javaservices/java/.../不存在，由 OH foundation 管理

构建目标 services 完全不需要编译。

5.2 系统应用（packages/）

所有预装系统应用均由 OH 对应应用替代：

  • packages/apps/Launcher3/ → OH Launcher
  • frameworks/base/packages/SystemUI/ → OH SystemUI
  • packages/apps/Settings/ → OH Settings
  • packages/apps/Contacts/ → OH Contacts
  • packages/apps/Dialer/ → OH Dialer
  • 其他所有 packages/apps/*

5.3 HAL 层（hardware/）

Android HAL（Hardware Abstraction Layer）由 OH 的 HDI（Hardware Device Interface）替代。包括：

  • Graphics HAL (gralloc, HWC) → OH Graphics HDI
  • Audio HAL → OH Audio HDI
  • Camera HAL → OH Camera HDI
  • Sensors HAL → OH Sensors HDI

不需要编译 hardware/ 目录下的任何内容。

6.3 app_process 与 appspawn-x 的关系

标准 AOSP 中，App 进程由 Zygote fork 产生，入口为 app_process64（frameworks/base/cmds/app_process/app_main.cpp）。在适配项目中：

  • app_process64 被 appspawn-x 替代
  • appspawn-x 的 ART 初始化逻辑参考了 app_main.cpp 中的 AndroidRuntime::start()
  • app_process64 可作为可选编译目标，用于调试和对比验证
  • appspawn-x 本身通过 OH 构建系统（BUILD.gn）编译，不通过 AOSP 构建

7. 构建策略

7.1 方案 A：模块级按需编译

使用 AOSP 标准构建系统，仅编译所需模块：

# 初始化构建环境
source build/envsetup.sh
lunch aosp_arm64-eng

# 编译核心模块（依赖自动解析）
m framework-minus-apex     # → framework.jar (客户端 Framework)
m libandroid_runtime       # → libandroid_runtime.so (JNI 桥接)
m framework-res            # → framework-res.apk (系统资源)
m dex2oat                  # → dex2oat (AOT 编译器)

# 适配层模块
m liboh_adapter_bridge     # → liboh_adapter_bridge.so
m oh-adapter-framework     # → oh-adapter-framework.jar

# ART / bionic / libcore 作为依赖自动编译，无需显式指定

优点

  • 简单直接，利用 AOSP 现有依赖解析
  • 无需创建自定义 makefile
  • 可快速验证

缺点

  • 首次编译仍需较长时间（依赖链较长）
  • 可能拉入不必要的依赖模块
  • lunch aosp_arm64-eng 配置了完整设备产品，部分配置可能不适用

7.2 方案 B：自定义 Product

创建最小化的 Product makefile，精确控制编译范围：

# device/adapter/oh_adapter/AndroidProducts.mk
PRODUCT_MAKEFILES := \
    $(LOCAL_DIR)/oh_adapter.mk

COMMON_LUNCH_CHOICES := \
    oh_adapter-eng

# device/adapter/oh_adapter/oh_adapter.mk
$(call inherit-product, build/target/product/core_64_bit_only.mk)

PRODUCT_NAME := oh_adapter
PRODUCT_DEVICE := generic_arm64
PRODUCT_BRAND := Android

# 仅包含 App 运行时必需产物
PRODUCT_PACKAGES += \
    framework \
    framework-res \
    libandroid_runtime \
    core-oj \
    core-libart \
    libart \
    dex2oat \
    oh-adapter-framework \
    liboh_adapter_bridge

# 不包含系统服务和系统应用
PRODUCT_PACKAGES_REMOVE += \
    SystemUI \
    Launcher3 \
    Settings

优点

  • 精确控制编译范围，排除不需要的模块
  • 编译速度更快（减少不必要的构建）
  • 适合长期维护
  • 可定义自己的 lunch target

缺点

  • 需要理解 AOSP Product 构建体系
  • 初始配置工作量较大
  • 需要处理依赖链中的隐式产品配置

7.3 方案对比与建议

维度方案 A（按需编译）方案 B（自定义 Product）

编译速度中（可能拉入额外依赖）快（精确控制范围）
可维护性低（无集中配置）高（产品定义集中管理）
可重复性中（依赖手动指定目标）高（lunch + make 即可）
适用阶段早期验证持续开发
建议：先用方案 A 快速验证编译流程和产物完整性，确认无误后再迁移到方案 B 建立正式的构建配置。

9. 结论

核心结论

  1. AOSP 构建体系天然支持客户端/服务端分离 — framework.jar（客户端）和 services.jar（服务端）是完全独立的构建目标，无需额外裁剪即可只编译客户端部分。
  2. 编译范围明确 — 需要编译 9 个核心模块（Framework 客户端、ART、libcore、bionic、原生框架库、系统基础库、JNI 辅助库、适配层、预编译工具链），排除系统服务、系统应用和 HAL 层。
  3. AOSP 修改点集中 — 4 个源文件修改全部位于 frameworks/base/core/java/ 内，属于必须编译的模块范围。
  4. 服务器环境就绪 — oh-build 服务器（16 核、60GB 内存、378GB 可用磁盘）完全满足编译需求。
  5. 建议分两步走 — 先用方案 A（模块级按需编译）快速验证，再迁移到方案 B（自定义 Product）建立正式构建流程。

10.5 经验总结

  1. 部分同步编译 AOSP 的核心挑战不在于构建目标选择，而在于 Soong 全局扫描 — Soong 会解析源码树中所有 Android.bp 文件并生成完整的 build.ninja，即使只编译单个目标。缺失的仓库会导致模块定义错误、nil pointer panic、依赖循环等问题。
  2. ALLOW_MISSING_DEPENDENCIES=true 仅在 ninja 阶段生效 — 它不能解决 Soong build.ninja 生成阶段的错误（如缺失模块类型、APEX panic、文件不存在等）。这些必须通过同步缺失仓库或禁用有问题的 bp 文件来解决。
  3. 模块 stubs 是 framework.jar 编译的硬依赖 — framework-minus-apex 虽然只包含非可更新的客户端代码，但其源码中大量引用了可更新模块的 API（如 android.net.*、android.provider.MediaStore、android.health.connect.* 等）。这些类定义来自各模块的 stubs.module_lib jar，必须出现在编译 classpath 上。
  4. API 兼容性检查是最大的阻碍之一 — check_api、api_lint、check_last_released_api 在部分同步环境下极易失败（缺少 baseline 文件、API 变更检测误报等）。对于开发构建，全部禁用是最高效的做法。
  5. 建议采用"渐进式同步"策略 — 从最小集开始编译，根据错误信息按需同步缺失仓库，而非一次性同步完整 AOSP（~300GB）。最终同步了 80+ 个仓库（约 132GB），远小于完整 AOSP。

10.8 编译产物可部署性分析：Java 层 vs Native 层

核心结论：当前 AOSP 编译产物不能直接运行在 OH 系统上。Java/DEX 层（framework.jar）可直接部署，但 Native 层（.so）必须用 OH 工具链交叉重编。

问题本质：Bionic vs musl

Android 和 OpenHarmony 底层都是 Linux 内核 + ARM64 指令集，机器码本身是兼容的。但两者的用户态 ABI 不同——C 库符号表、动态链接器、TLS 机制各异，导致 Native 二进制不能互通：

维度AOSP 编译产物OH 系统环境是否兼容
目标三元组aarch64-linux-androidaarch64-linux-ohos❌ 不兼容
C 标准库Bionic libc（libc.so）musl libc（libc.so）❌ 符号表不同
动态链接器/system/bin/linker64/system/lib/ld-musl-aarch64.so.1❌ 完全不同
编译器Android ClangOH Clang❌ sysroot 不同
Java/DEX 字节码classes.dex（平台无关）由 ART VM 解释/JIT✅ 兼容

分层可部署性

层次产物能否直接部署到 OH说明
Java/DEX framework.jar ✅ 可以 DEX 字节码与平台无关，在任何 ART VM 实例上都能运行
oh-adapter-framework.jar ✅ 可以 适配层 Java 代码，同为 DEX 字节码
framework-res-package.jar ✅ 可以 编译后的资源二进制格式，平台无关
AOSP Native libart.so ❌ 不能 链接 Bionic libc，OH 上无法被 musl 动态链接器加载
libandroid_runtime.so ❌ 不能 依赖 libart + Bionic，同上
libbinder.so, libgui.so ❌ 不能 链接 Bionic + Android 内核特有 Binder 驱动接口
linker64, libc.so ❌ 不能 Android 动态链接器/C 库，与 OH 的 musl 冲突
OH Native liboh_adapter_bridge.so ✅（需 OH 编译） 必须用 OH 工具链（BUILD.gn）编译，链接 OH IPC 库
appspawn-x ✅（需 OH 编译） 必须用 OH 工具链（BUILD.gn）编译，为 OH init 服务

当前 AOSP 编译产物的价值

  1. Java 层可直接部署 — framework.jar（39MB）包含完整的客户端 Framework DEX 字节码，可直接放入 OH 上运行的 ART VM 环境
  2. 验证了最小化构建可行性 — 确认 AOSP 14 可以只编译客户端 JAR，不需要 system_server 和系统应用
  3. 为交叉编译提供了源码基线 — 后续用 OH Clang 工具链重编 Native 层时，使用同一套已验证的 AOSP 源码
  4. 确定了依赖范围 — 明确了需要同步的 115 个 AOSP 仓库和所有编译配置修改

完整运行还需要的交叉编译产物

以下产物必须使用 OH Clang 工具链（aarch64-linux-ohos target）交叉编译 AOSP C++ 源码，使其链接 musl libc：

产物AOSP 源码编译方式核心难点
libart.so art/runtime/ OH Clang 交叉编译 Bionic 专有 API 适配（system_properties、malloc、TLS 等）
libandroid_runtime.so frameworks/base/core/jni/ OH Clang 交叉编译 依赖 libart + Surface/Input JNI 桥接
dex2oat art/dex2oat/ OH Clang 交叉编译 依赖 libart + libdexfile
bionic_compat 兼容层 新建适配代码 OH Clang 编译 将 Bionic 专有 API 转调到 OH API（sysparam 替代 system_properties 等）
liboh_adapter_bridge.so framework/*/jni/ OH BUILD.gn 链接 OH IPC 库（libabilityms_proxy、libwms_proxy 等）
appspawn-x framework/appspawn-x/ OH BUILD.gn 混合进程启动器，集成 OH 沙箱 + ART 初始化

交叉编译的详细方案见 appspawn_x_design.html 第 6.5 节《Android 运行时库交叉编译》。核心思路是：用 OH Clang 重编 AOSP C++ 源码，将 Bionic 专有 API（system_properties、android_mallopt 等）替换为 OH 等效实现（SystemGetParameter、musl malloc 等）。
